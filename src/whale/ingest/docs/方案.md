# ingest 模块完整方案（优化版）

## 一、设计目标

当前 `ingest` 模块的目标，不应绑定到某一种协议或某一种访问方式，而应抽象为：

1. 从多种工业数据源持续获取数据  
2. 将数据收敛为平台内部统一的“当前状态”  
3. 按统一规则输出到下游管道  
4. 支持不同采集模式：
   - 订阅
   - 轮询
   - 批次读取
   - 后续可扩展文件扫描、消息消费等
5. 保持 clean 风格分层，同时允许 use case 内部采用 DCI 思想组织协作

因此，`ingest` 不应围绕 “OPC UA / Modbus / MMS” 来建 use case，  
也不应围绕 “订阅 / 轮询” 直接建第一层顶级 use case。

更稳妥的边界是：

- **MaintainSourceStateUseCase**：负责把外部源数据收敛到内部状态
- **EmitStateSnapshotUseCase**：负责把内部状态稳定输出到下游

协议、设备类型、访问方式，都属于这些 use case 的实现策略差异，而不是顶层业务目标本身。

---

## 二、总体分层结论

你当前目录结构是：

- `adapters`
- `ports`
- `usecases`
- `framework`

这个结构可以继续用，不必再加新顶层目录。

建议职责固定如下：

### 1. `usecases`
放“应用层用例”和用例上下文对象。

它们表达业务目标，不表达具体协议细节。

这里的 use case 可以使用 DCI 思想：
- use case = context
- context 内协作角色 = role
- role 所作用的对象 = state/entity/runtime object

但目录命名仍然用 `usecases`，不要改成 `contexts`，否则工程语义会变散。

---

### 2. `ports`
放应用层定义的抽象边界接口。

这些接口定义：
- use case 需要什么能力
- 外部实现必须提供什么契约

`ports` 不放到 `usecases` 子目录内，必须单独存在。  
因为 port 是应用边界，不是某个用例私有细节。

---

### 3. `adapters`
放对 `ports` 的具体实现，负责协议对接、对象转换、外部系统接入。

例如：
- OPC UA 订阅适配器
- OPC UA 轮询适配器
- 消息管道发布适配器
- 内存状态存储适配器

---

### 4. `framework`
放最外层运行支撑与应用装配逻辑。

包括：
- 启动入口
- 顶层 orchestrator
- app builder
- runtime loop / scheduler / cancellation
- 配置装配
- 顶层异常管理

`framework` 不放业务规则，不放协议解析，不放领域状态逻辑。  
它只做“把系统跑起来”。

---

## 三、核心架构思想

### 1. 顶层按“稳定目标”划分 use case

建议固定为两个主 use case：

#### A. `MaintainSourceStateUseCase`
职责：

- 启动数据源采集
- 建立 source session
- 根据源能力执行订阅/轮询/批次读取
- 把采集结果规范化为内部采集事件
- 更新状态缓存
- 更新 freshness / quality / health
- 在异常时执行恢复

它的输出不是“发下游”，而是：

> 维护一份可信的当前状态

---

#### B. `EmitStateSnapshotUseCase`
职责：

- 周期读取状态缓存
- 构造平台内部统一快照
- 打上时间戳、质量位、stale 标记
- 发布到下游消息管道
- 维护输出节拍与发布重试

它不关心上游到底是订阅来的、轮询来的，还是批次读取来的。

---

### 2. 访问模式不是顶层用例，而是 source acquisition strategy

例如：

- 订阅
- 轮询
- 批读

这些属于：

- source runtime strategy
- source adapter capability
- acquisition role

而不是顶层 use case 名称的一部分。

也就是说，不推荐建这些顶层用例：

- `SubscribeOpcUaUseCase`
- `PollModbusUseCase`
- `BatchReadMmsUseCase`

因为这会导致：
- 协议与模式耦合进用例边界
- 后续设备类型增加时爆炸
- 用例层沦为技术动作枚举

---

### 3. 状态缓存是两个 use case 的中间边界

这是整个方案的关键。

`MaintainSourceStateUseCase` 负责写入状态  
`EmitStateSnapshotUseCase` 负责读取状态

所以必须有一个清晰的共享边界：

- `PointStateStorePort`

它的实现首版建议用内存，不要一开始上数据库。

数据库更适合：
- 审计
- 回放
- 落库
- 恢复

不适合作为首版实时状态热路径的核心依赖。

---

## 五、每个代码文件职责

---

### A. 根入口

#### `ingest/__main__.py`
职责：

- 作为包启动入口
- 调用 `framework.orchestrator` 或 `framework.builder` 提供的 `main()`
- 不承载业务逻辑

典型启动方式：

    PYTHONPATH=src python -m whale.ingest

这个文件要非常薄，只做顶层转发。

---

### B. `ports/` 层

---

#### `ports/source_ports.py`
职责：

定义数据源采集侧的抽象能力接口。

建议放这些抽象：

- `SourceSessionPort`
  - 建立/关闭会话
  - 查询健康状态
  - 执行重连

- `SubscribableSourcePort`
  - 启动订阅
  - 停止订阅
  - 注册通知回调或事件输出

- `PollableSourcePort`
  - 执行一次轮询读取
  - 返回当前一批采集结果

- `BatchReadableSourcePort`
  - 批量读取指定点位快照

- `SourceCapabilityDescriptor`
  - 描述某个 source 支持哪些访问模式

说明：

这里不直接出现“OPC UA”或“Modbus”名字。  
因为这层表达的是能力抽象，不是协议实现。

---

#### `ports/store_ports.py`
职责：

定义状态缓存边界。

建议放：

- `PointStateStorePort`
  - upsert 单点状态
  - batch_upsert 批量状态
  - get_point
  - get_snapshot
  - mark_stale
  - update_source_health

- `StateSnapshotView`
  - 表示一次读取出来的完整状态快照视图

这一层是两个 use case 的中间契约核心。

---

#### `ports/sink_ports.py`
职责：

定义下游输出边界。

建议放：

- `SnapshotPublisherPort`
  - 发布一次快照消息

- `PublishResult`
  - 表示发布结果

---

#### `ports/runtime_ports.py`
职责：

定义运行支撑能力接口。

建议放：

- `SchedulerPort`
- `ClockPort`
- `CancellationPort`
- `LifecycleSignalPort`

说明：

这层只抽象 runtime 能力，不抽象业务。

---

### C. `usecases/` 层

---

#### `usecases/models.py`
职责：

放 use case 层使用的运行模型，不放协议细节。

建议放：

- `SourceDefinition`
  - 设备源定义
  - 协议类型、点位集、访问模式、连接配置引用

- `AcquisitionMode`
  - `SUBSCRIPTION`
  - `POLLING`
  - `BATCH_READ`

- `NormalizedPointEvent`
  - 统一后的内部采集事件
  - 点标识、值、时间戳、质量位、来源信息

- `PointState`
  - 当前单点状态
  - 最新值、source_ts、recv_ts、quality、stale

- `SourceHealthState`
  - 当前源健康状态
  - running / degraded / disconnected / recovering

- `SnapshotEnvelope`
  - 输出到下游前的统一快照对象

这里是应用层运行对象，不直接依赖协议库。

---

#### `usecases/policies.py`
职责：

放用例层策略与规则对象。

建议放：

- `StalenessPolicy`
  - 多久未更新视为 stale

- `SnapshotEmissionPolicy`
  - 多久发一次快照
  - 是否允许空快照
  - 是否附带 health

- `RecoveryPolicy`
  - 重试次数
  - 退避间隔
  - 何时宣布失败

- `SourceSelectionPolicy`
  - 根据 source capability 选择订阅/轮询/批读策略

这些是应用层业务规则，不能放到 adapter 或 framework。

---

#### `usecases/roles.py`
职责：

放 DCI 风格的角色定义。

建议只放“角色协议”或“角色协作辅助对象”，不要放具体协议实现。

可包括：

- `SourceSessionRole`
- `AcquisitionRole`
- `StateUpdateRole`
- `HealthCheckRole`
- `RecoveryRole`
- `SnapshotBuildRole`
- `SnapshotPublishRole`

说明：

这里的 role 是 use case 内部的协作责任抽象。  
不要把 role 写成和协议绑定的类名，例如：
- 不要 `OpcUaSubscriptionRole`
- 不要 `ModbusPollingRole`

协议差异应由 adapter 或 strategy 注入。

---

#### `usecases/maintain_source_state.py`
职责：

实现第一个主 use case：

> 持续采集并维护当前源状态

它是整个 ingest 模块的核心之一。

建议职责包括：

1. 接收 `SourceDefinition`
2. 基于 source capability 和 policy 选择 acquisition mode
3. 建立 source session
4. 启动订阅或轮询流程
5. 把外部输入映射成 `NormalizedPointEvent`
6. 更新 `PointStateStorePort`
7. 更新 `SourceHealthState`
8. 在异常时执行恢复流程

这个文件里允许使用 DCI 方式组织，例如：

- 一个主 context 类
- 多个 role 协作方法
- 不同 mode 的执行分支

但要注意：

- 它是 use case
- 不是协议 SDK 包装层
- 不是线程/async event loop 管理层

---

#### `usecases/emit_state_snapshot.py`
职责：

实现第二个主 use case：

> 周期从状态缓存生成完整快照并输出到下游

建议职责包括：

1. 根据 `SnapshotEmissionPolicy` 定时触发
2. 从 `PointStateStorePort` 获取当前快照
3. 附加 source health / stale / timestamps
4. 生成 `SnapshotEnvelope`
5. 调用 `SnapshotPublisherPort`
6. 处理发布失败、重试、日志

它不感知 source 是怎么来的。  
它的输入是 store，它的输出是 sink。

---

### D. `adapters/` 层

---

#### `adapters/source/opcua/subscription_source_adapter.py`
职责：

实现 `SubscribableSourcePort`。

负责：

- 建立 OPC UA 订阅
- 注册 monitored items
- 接收通知
- 交付给 use case 需要的统一事件出口

注意：

这里不负责更新 `PointStateStorePort`，  
那是 use case 的职责。

它最多只负责把协议通知转换成较薄的 adapter event。

---

#### `adapters/source/opcua/polling_source_adapter.py`
职责：

实现 `PollableSourcePort` / `BatchReadableSourcePort`。

负责：

- 建立 OPC UA 会话
- 执行周期性读取或批量读取
- 返回当前读取结果

说明：

你前面分析得对，很多设备场景下轮询/批次读取反而更简单。  
所以这个 adapter 是必须保留的，不应该因为“当前主推订阅”就删掉。

---

#### `adapters/source/opcua/opcua_event_mapper.py`
职责：

把 OPC UA SDK 返回的数据结构映射为内部统一事件/读取结果。

负责：

- value/status/timestamp 映射
- 节点标识到内部 point_id 映射
- 质量位转换

这个文件的作用是切断 use case 对 asyncua 之类 SDK 结构的依赖。

---

#### `adapters/source/opcua/opcua_connection_factory.py`
职责：

集中处理 OPC UA 连接对象创建、配置注入、会话构建。

负责：

- endpoint
- security
- session options
- timeout
- reconnect primitive

它是 adapter 内部基础设施辅助件，不属于 use case。

---

#### `adapters/source/modbus/polling_source_adapter.py`
职责：

提供 Modbus 风格的轮询实现。

即使当前未实现，也应留出这个位置。  
这样可以逼迫架构从一开始就不把 `ingest` 设计成 “仅 OPC UA ingest”。

---

#### `adapters/source/common/source_runtime_models.py`
职责：

定义 adapter 内部共享的 source runtime 数据对象。

例如：

- `SourceReadResult`
- `SourceNotification`
- `ConnectionState`
- `SubscriptionHandle`

这些对象属于 adapter 内部，不进入 use case 核心模型。

---

#### `adapters/store/in_memory_point_state_store.py`
职责：

实现 `PointStateStorePort` 的内存版本。

负责：

- 维护最新点位状态表
- 支持并发更新与快照读取
- 支持 source health 状态维护

这是首版推荐的默认 store 实现。

不要一开始把它做成数据库驱动。  
数据库版本后续可作为另一个 adapter 再扩。

---

#### `adapters/sink/message_pipeline_publisher.py`
职责：

实现 `SnapshotPublisherPort`。

负责：

- 把 `SnapshotEnvelope` 发布到消息管道
- 处理下游发送错误
- 返回发布结果

---

#### `adapters/sink/publisher_message_mapper.py`
职责：

把内部 `SnapshotEnvelope` 转为消息管道需要的外部消息格式。

例如：

- JSON payload
- topic / key / headers
- message schema version

这个映射不要放到 use case 里。

---

### E. `framework/` 层

---

#### `framework/config.py`
职责：

集中处理 ingest 模块运行配置。

例如：

- source definitions
- emission interval
- reconnect policy
- pipeline config

说明：

这里是运行装配配置，不是 use case policy 定义本身。

---

#### `framework/builder.py`
职责：

应用装配器。

负责：

1. 读取配置
2. 创建 store adapter
3. 创建 source adapter
4. 创建 sink adapter
5. 创建 use case
6. 创建 orchestrator
7. 返回可运行应用对象

它是整个模块的 composition root。

---

#### `framework/orchestrator.py`
职责：

顶层运行编排器。

负责：

- 启动 `MaintainSourceStateUseCase`
- 启动 `EmitStateSnapshotUseCase`
- 管理生命周期
- 处理停止顺序
- 传播 cancellation
- 顶层异常收敛

注意：

它不应实现业务规则，  
只应管理运行单元的启动、等待、关闭。

---

#### `framework/runtime.py`
职责：

运行时基础支撑。

负责：

- async task group
- 周期调度
- background loop
- runtime helpers

如果你后面不用 async，也可以保留这里做线程/任务调度抽象。

---

#### `framework/shutdown.py`
职责：

处理停机流程。

负责：

- signal handler
- graceful shutdown
- flush / close / cancel

这样能避免 `orchestrator.py` 过大。

---

## 六、关键相互关系

整体依赖方向应为：

    framework -> usecases -> ports
    adapters  -> ports

说明：

- `framework` 负责装配 use case 和 adapter
- `usecases` 依赖 `ports`
- `adapters` 实现 `ports`
- `usecases` 不能反向依赖 `adapters`
- `ports` 不能依赖 `adapters`
- `framework` 可以引用所有外层对象进行装配，但不承载业务规则

---

### 启动关系

启动时的关系建议是：

1. `__main__.py` 启动
2. 进入 `framework.builder`
3. builder 创建：
   - `PointStateStorePort` 实现
   - source adapter
   - sink adapter
   - 两个 use case
   - orchestrator
4. orchestrator 启动：
   - `MaintainSourceStateUseCase`
   - `EmitStateSnapshotUseCase`
5. 二者通过 `PointStateStorePort` 共享状态

---

### 运行关系

#### `MaintainSourceStateUseCase`
依赖：

- `SourceSessionPort`
- `SubscribableSourcePort` 或 `PollableSourcePort`
- `PointStateStorePort`
- `ClockPort`
- `RecoveryPolicy`
- `StalenessPolicy`

输出：

- 更新 store
- 更新 source health

---

#### `EmitStateSnapshotUseCase`
依赖：

- `PointStateStorePort`
- `SnapshotPublisherPort`
- `ClockPort`
- `SnapshotEmissionPolicy`

输出：

- 向消息管道发快照

---

## 七、DCI 的具体使用建议

可以用，但要收敛在 `usecases/` 内。

建议模式：

### 1. 把主用例写成 Context
例如：

- `MaintainSourceStateContext`
- `EmitStateSnapshotContext`

### 2. 用 role 表达内部协作责任
例如：

- `AcquisitionRole`
- `StateUpdateRole`
- `RecoveryRole`
- `SnapshotBuildRole`

### 3. role 不直接感知协议实现
协议差异通过 port 和 adapter 注入。

### 4. 不要把整个工程都 role 化
只在复杂 use case 内用 DCI。
`adapters`、`framework` 不需要做成 DCI。

---

## 八、关于“订阅一种 use case，轮询一种 use case，还是统一”的最终建议

结论是：

### 顶层 use case 统一
统一成：

- `MaintainSourceStateUseCase`
- `EmitStateSnapshotUseCase`

### 获取方式作为 mode strategy 分流
例如：

- subscription mode
- polling mode
- batch-read mode

### 协议作为 adapter 分流
例如：

- opcua adapter
- modbus adapter
- mms adapter

所以最终是一个三层分离：

1. **业务目标层**：use case  
2. **访问模式层**：strategy / mode  
3. **协议实现层**：adapter  

这是最稳的。

---

## 九、为什么这个方案比“按协议或按模式拆 use case”更好

### 1. 业务边界稳定
设备和协议会变，  
“维护状态”和“输出快照”不会变。

### 2. 扩展成本低
以后新增：
- Modbus
- IEC 104
- MMS
- 文件扫描

只需要新增 adapter 和部分 mode strategy，  
不需要重做顶层 use case 体系。

### 3. 测试更清晰
可以分别测：

- use case 是否正确使用 port
- adapter 是否正确实现协议接入
- framework 是否正确装配启动

### 4. 不会被当前 scenario1 锁死
虽然现在你可能主要做某个单场景，但 ingest 作为基础模块，不能被单协议单模式绑死。

---

## 十、最小落地顺序建议

为了避免一上来过度设计，建议按以下顺序落地：

### 第一步
先搭出骨架文件：

- `__main__.py`
- `ports/source_ports.py`
- `ports/store_ports.py`
- `ports/sink_ports.py`
- `usecases/models.py`
- `usecases/policies.py`

## 十一、最终建议摘要

### 顶层目录继续保留
- `adapters`
- `ports`
- `usecases`
- `framework`

### use case 用 DCI
可以，但只在 `usecases/` 内部使用。

### ports 单独目录
不要放进 `usecases/`。

### 顶层 use case 不按协议分
推荐固定为：
- `MaintainSourceStateUseCase`
- `EmitStateSnapshotUseCase`

### 订阅/轮询不是顶层 use case
它们是 acquisition mode / strategy。

### 协议不是顶层 use case
协议放在 `adapters` 层实现。

### 缓存是核心中间边界
通过 `PointStateStorePort` 解耦接入与输出。

### framework 只做装配和运行
不承载业务规则。

## 十二、一句话定版

`ingest` 模块应当被设计为：

> 一个以“状态收敛 + 快照输出”为核心的通用采集框架；  
> 顶层 use case 按稳定业务目标划分，  
> 访问模式按 strategy 划分，  
> 协议差异按 adapter 划分，  
> 中间通过 `PointStateStorePort` 解耦。
