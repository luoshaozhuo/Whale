# ingest 模块设计说明书

## 1. 平台总目标

Whale 的总体目标是建设一个面向并网型风电场、光伏电站、储能电站及风光储一体化电场（群）的统一数据基座。

这个统一数据基座需要承接现场运行、仿真分析、测试验证和后续数据治理的共同数据入口，最终形成一套可持续扩展的数据接入、存储、治理和分发底座，为下游数据接入、清洗、建模、分析、调度优化和业务应用提供统一上游。

从平台视角看，Whale 需要覆盖以下范围：

- 生产现场设备数据
- 场站级控制系统与子系统数据
- 仿真模型、仿真工况和仿真结果数据
- 测试数据、回放数据、联调数据
- 不同协议、不同文件格式、不同接入节奏的数据源

因此，平台总目标不是“接好某一个协议”，而是面向风光储电场（群）建设统一、可扩展、可演进的数据接入底座。

## 2. 平台模块划分与链路

从平台整体视角看，Whale 的主要模块划分如下。

### 2.1 Whale 自研程序模块

- `whale.ingest`
  采集模块，负责从外部数据源采集数据并输出到采集消息通道
- `whale.stream_ingestion`
  流式接入模块，负责从采集消息通道消费数据，并完成原始落库以及实时标准化、清洗、聚合
- `whale.batch_processing`
  批处理模块，负责从原始数据存储读取历史数据并执行离线处理
- `whale.serving`
  服务输出模块，负责面向上层应用提供查询、计算结果服务和消费接口

### 2.2 Whale 适配但不自研实现的外部基础设施

- `message-bus / ingestion-bus`
  采集消息总线，例如 Kafka、MQTT Broker、Redis Streams、NATS
- `raw-store`
  原始数据存储，例如 PostgreSQL、TimescaleDB、ClickHouse、MinIO 等

这些组件是 Whale 的外部依赖，Whale 负责接入、适配和使用它们，但不实现这些中间件或数据库本身。

### 2.3 平台总体处理链路

平台总体处理链路应理解为：

```text
外部数据源
  ↓
ingest（Whale 程序模块）
  ↓
message-bus / ingestion-bus（外部消息中间件）
  ↓
stream-ingestion（Whale 程序模块）
  ↓
raw-store（外部数据库 / 存储）
  ↓
batch-processing（Whale 程序模块）
  ↓
serving（Whale 程序模块）
```

这条链路的关键含义是：

- `ingest` 不是平台的全部，而是统一数据基座最前端的采集入口
- `message-bus / ingestion-bus` 是 `ingest` 与下游流式接入模块之间的标准解耦边界
- `stream-ingestion` 负责把采集消息进一步沉淀到 `raw-store`
- `batch-processing` 与 `serving` 位于采集之后的后续处理和服务阶段

## 3. ingest 模块定位

`whale.ingest` 是 Whale 统一数据基座中的采集入口模块。

它负责把外部数据源中的设备数据、仿真模型与仿真数据、测试数据等采集进来，统一转换为平台可消费的采集结果，并输出到采集消息通道或其他标准化边界，供下游模块继续消费、转换和处理。

从系统边界上说，`ingest` 面向的接入范围应当是宽的，而不是局限于单一协议或单一来源：

- 风电机组、光伏逆变器、箱变、SVG、储能变流器、电池管理系统、升压站设备等现场设备通信协议
- 激光雷达、测风塔、气象站、功率预测相关测风测辐照设备的数据接口
- 场站仿真模型和仿真结果文件
- 联调、验收、测试阶段的样例数据或批量数据文件
- 消息、文件、数据库等其他来源

在协议与数据来源层面，`ingest` 需要覆盖风光储电场常见接入方式，例如：

- OPC UA
- Modbus TCP / Modbus RTU
- IEC 60870-5-101 / 104
- IEC 61850
- MQTT
- HTTP / HTTPS API
- WebSocket
- 厂商私有 TCP/UDP 协议
- CSV、Excel、JSON、XML、YAML 等数据文件
- 仿真模型文件、仿真结果文件、回放文件、测试样例文件

也就是说，`ingest` 的定位是：

- 统一采集入口
- 统一采集调度与执行框架
- 统一采集结果输出边界

因此，这个模块的设计应当始终面向“大范围数据接入”，而不是围绕某一个已接入协议来收缩边界。

## 4. ingest 模块目标

### 4.1 总体目标

`ingest` 的总体目标是为 Whale 统一数据基座提供稳定、可扩展的数据采集能力，把多来源、多协议、多形态的数据收敛到统一入口，并可靠送入下游数据通道。

具体来说，`ingest` 应覆盖并网型风电场、光伏电站、储能电站及风光储一体化电场（群）中常见的数据来源，包括设备实时数据、仿真模型、仿真结果、测试数据和回放数据。

### 4.2 模块目标

`ingest` 的模块目标包括：

- 为已配置设备提供统一的运行时采集入口
- 支持 `ONCE`、`POLLING`、`SUBSCRIPTION` 三种采集模式
- 支持常见风光储场站协议和文件型数据源接入
- 支持风机、逆变器、储能设备、激光雷达、测风塔、气象设备等对象的数据采集
- 支持设备在线采集、仿真数据导入、测试数据导入等不同接入形态
- 将外部协议采集结果收敛为统一 DTO
- 刷新本地最新状态缓存，并在统一边界上组织完整状态快照
- 将完整状态快照发送到消息通道，供下游数据接入模块消费
- 在需要时维护最新变量状态视图，供本地运行时、调试和测试观察使用
- 为不同协议、不同源类型、不同输出方式保留统一扩展边界

## 5. 功能需求

### 5.1 配置与建库

- 支持初始化采集数据库结构
- 支持导入样例设备、模型、变量、任务数据
- 支持从数据库读取启用中的采集任务
- 支持根据任务、设备、模型、变量拼装运行时采集定义

### 5.2 运行时调度

- 支持按任务的 `acquisition_mode` 调度运行
- `ONCE` 任务执行一次后结束
- `POLLING` 任务按 `interval_ms` 周期执行
- `SUBSCRIPTION` 任务常驻运行，直到调度器停止
- 支持将同协议、同周期的轮询任务合并执行
- 支持将同协议的订阅任务合并为一个运行作业

### 5.3 数据采集

- 支持现场在线数据采集，包括主动读取和被动订阅两类基本方式
- 支持设备协议接入、文件导入、模型导入、测试数据导入等不同采集形态
- 支持按协议或来源类型扩展不同采集适配器
- 支持从 `host/port` 或完整 `endpoint` 解析连接地址
- 支持从 `namespace_uri` + 简写 locator 解析真实 node id

### 5.4 数据输出与状态存储

- 支持维护本地最新状态缓存，供本地运行时、调试观察和后续快照发射使用
- 支持从本地最新状态缓存读取完整状态快照
- 支持将完整状态快照组装为统一消息结构
- 支持通过文件、Redis Streams、Kafka 等发布适配器发送完整状态快照
- 支持按统一结构保存源时间、接收时间、本地更新时间等关键元数据
- 支持在测试环境中提供可观察、可核对的落盘输出

## 6. 非功能需求

### 6.1 可维护性

- 采用 `ports / adapters / usecases / framework` 分层
- 业务用例不直接依赖具体数据库驱动或协议库
- 协议接入和存储实现通过 port 边界隔离

### 6.2 可测试性

- 核心 use case 有独立单元测试
- OPC UA 仿真器支持集成测试
- 提供测试专用 `ModeCaptureStateCache`（位于 tests helper），便于观察采集输出

### 6.3 可扩展性

- 采集协议通过 `SourceAcquisitionPort` 和 registry 扩展
- 本地状态缓存通过 `SourceStateCachePort` 扩展
- 完整快照读取通过 `SourceStateSnapshotReaderPort` 扩展
- 快照消息发布通过 `MessagePublisherPort` 扩展
- 调度模式已经预留 `ONCE / POLLING / SUBSCRIPTION` 三类基础形态

### 6.4 运行稳定性

- 调度器支持统一停止控制
- 订阅作业通过 `stop_event` 停止
- 主入口在退出时主动释放数据库连接池

## 7. 架构说明

### 7.1 总体架构原则

`ingest` 当前总体逻辑架构以 Clean Architecture 为总原则。

在这个总原则下，当前代码有两个关键落地特征：

- 在外部边界实现上，采用 Port-Adapter 方式隔离协议、数据库和文件系统等外部依赖
- 在应用层内部协作上，采用 CRI 风格组织 use case 与角色对象之间的协作关系

因此，更严谨的表达不是”Clean Architecture、Port-Adapter、CRI 三者并列”，而是：

- Clean Architecture 规定整体分层和依赖方向
- Port-Adapter 是 Clean Architecture 在边界上的具体落地方式
- CRI 是 Clean Architecture 应用层内部的一种协作组织方式

从代码实际落地情况看，这三者的关系分别体现在以下几个方面。

#### 7.1.1 Clean Architecture 分层

当前模块围绕应用层边界进行组织，核心目录为：

- `usecases`
- `ports`
- `adapters`
- `framework`
- `runtime`

其中：

- `usecases` 负责表达应用层动作，例如拉取采集和订阅采集
- `ports` 定义应用层对外部能力的抽象依赖
- `adapters` 负责把数据库、协议库、文件输出等外部实现接到 port 上
- `framework` 负责数据库与初始化等基础设施装配
- `runtime` 负责调度器和运行控制

这种结构体现的是”业务动作依赖抽象边界，外部实现通过适配器注入”的 Clean Architecture 思路，而不是把协议细节、数据库访问和用例逻辑直接混写在一起。

#### 7.1.2 use case 内部采用 CRI 风格

当前 `ingest` 的 use case 没有把所有动作堆进一个大函数里，而是通过 `roles/` 目录把用例协作拆成更清晰的角色对象。

从代码现状看：

- [PullSourceStateUseCase](/home/luosh/Whale/src/whale/ingest/usecases/pull_source_state_usecase.py:1) 负责组织一次批量拉取流程
- [SubscribeSourceStateUseCase](/home/luosh/Whale/src/whale/ingest/usecases/subscribe_source_state_usecase.py:1) 负责组织长运行订阅流程
- [EmitStateSnapshotUseCase](/home/luosh/Whale/src/whale/ingest/usecases/emit_state_snapshot_usecase.py:1) 负责读取本地完整状态快照、组装快照消息并发布
- [PullRole](/home/luosh/Whale/src/whale/ingest/usecases/roles/pull_role.py:1) 负责构造采集请求并执行读取
- [SubscribeRole](/home/luosh/Whale/src/whale/ingest/usecases/roles/subscribe_role.py:1) 负责构造订阅请求并处理订阅回调协作
- [StateUpdateRole](/home/luosh/Whale/src/whale/ingest/usecases/roles/state_update_role.py:1) 负责把采集结果刷新到本地状态缓存
- [StateSnapshotReadRole](/home/luosh/Whale/src/whale/ingest/usecases/roles/state_snapshot_read_role.py:1) 负责读取当前完整缓存快照
- [StateSnapshotPublishRole](/home/luosh/Whale/src/whale/ingest/usecases/roles/state_snapshot_publish_role.py:1) 负责把快照消息通过多个 publisher 逐个发布
- [StateSnapshotMessageAssembler](/home/luosh/Whale/src/whale/ingest/usecases/roles/state_snapshot_message_assembler.py:1) 负责把缓存快照组装为统一消息 DTO

这类组织方式可以概括为：

- Context：use case 作为流程上下文
- Role：角色对象承载协作职责
- Interaction：由 use case 编排角色之间的调用关系

因此，这里更准确的说法不是”纯脚本式应用服务”，而是”在 Clean Architecture 应用层内部，用 CRI 风格细化用例协作”。

#### 7.1.3 Port-Adapter 作为边界落地方式

`ingest` 并不是把协议实现直接写死在 use case 里，而是通过 port 定义能力边界，再由 adapter 提供具体实现。

当前代码中的典型例子：

- `SourceAcquisitionPort` 定义采集能力边界
- `SourceAcquisitionDefinitionPort` 定义采集定义装配边界
- `SourceStateCachePort` 定义本地最新状态缓存写入边界
- `SourceStateSnapshotReaderPort` 定义完整状态快照读取边界
- `MessagePublisherPort` 定义快照消息发布边界
- [OpcUaSourceAcquisitionAdapter](/home/luosh/Whale/src/whale/ingest/adapters/source/opcua_source_acquisition_adapter.py:1) 是 `SourceAcquisitionPort` 的一个具体实现
- [OpcUaSourceAcquisitionDefinitionRepository](/home/luosh/Whale/src/whale/ingest/adapters/config/opcua_source_acquisition_definition_repository.py:1) 是采集定义 port 的一个具体实现
- [SqliteSourceStateCache](/home/luosh/Whale/src/whale/ingest/adapters/state/sqlite_source_state_cache.py:1) 同时实现本地状态缓存写入和完整快照读取
- [RedisSourceStateCache](/home/luosh/Whale/src/whale/ingest/adapters/state/redis_source_state_cache.py:1) 是生产推荐的近端最新状态缓存实现
- [RelationalOutboxMessagePublisher](/home/luosh/Whale/src/whale/ingest/adapters/message/relational_outbox_message_publisher.py:1)、[RedisStreamsMessagePublisher](/home/luosh/Whale/src/whale/ingest/adapters/message/redis_streams_message_publisher.py:1)、[KafkaMessagePublisher](/home/luosh/Whale/src/whale/ingest/adapters/message/kafka_message_publisher.py:1) 是三种快照消息发布实现

这意味着后续扩展 Modbus、IEC 61850、IEC 104、MQTT、文件导入等能力时，原则上应继续沿着同一模式扩展：

- 在 `ports` 中定义边界
- 在 `adapters` 中增加协议或文件型实现
- 在 `usecases` 中继续依赖抽象 port，而不是依赖某个具体协议类

### 7.2 当前实现形态与开发环境

#### 开发基础设施（Docker Compose）

本地开发通过 `docker-compose.ingest-dev.yaml` 一键启动完整基础设施栈：

- **PostgreSQL 16**：采集数据库，端口 `5432`
- **Redis 7**：状态缓存 + 可选消息通道，宿主机端口 `16379`
- **Kafka**：采集消息总线，端口 `9092`

环境变量统一由 `.env.ingest.example` 管理（`scripts/run_ingest_dev.sh` 直接加载），默认 backend 组合为 `postgresql + redis + kafka`。

#### 运行时装配

在上述总体原则之下，当前代码实现可以概括为一个”数据库驱动配置 + 调度器驱动运行 + OPC UA adapter 执行采集 + 本地状态缓存 + 独立快照发布”的结构。这是当前已经落地的实现形态，不等同于 `ingest` 的完整目标边界。

核心调用链如下：

1. `python -m whale.ingest` 进入 [__main__.py](/home/luosh/Whale/src/whale/ingest/__main__.py:1)
2. `config.py` 读取 `WHALE_INGEST_DATABASE_BACKEND` / `WHALE_INGEST_STATE_CACHE_BACKEND` / `WHALE_INGEST_MESSAGE_BACKEND` 三个环境变量选择后端
3. 组装 `SourceRuntimeConfigRepository`、`OpcUaSourceAcquisitionDefinitionRepository`、`OpcUaSourceAcquisitionAdapter`，并按环境选择 `SqliteSourceStateCache` 或 `RedisSourceStateCache`
4. 创建 [SourceScheduler](/home/luosh/Whale/src/whale/ingest/runtime/scheduler.py:1)
5. 调度器读取启用任务并按模式注册作业
6. 作业调用 [PullSourceStateUseCase](/home/luosh/Whale/src/whale/ingest/usecases/pull_source_state_usecase.py:1) 或 [SubscribeSourceStateUseCase](/home/luosh/Whale/src/whale/ingest/usecases/subscribe_source_state_usecase.py:1)
7. use case 通过配置仓储构造采集定义，通过 source adapter 执行采集
8. `SUBSCRIPTION` 作业在正式订阅前先做一次全量读取，初始化本地状态缓存
9. 采集结果通过 state cache adapter 刷新到本地最新状态缓存
10. [EmitStateSnapshotUseCase](/home/luosh/Whale/src/whale/ingest/usecases/emit_state_snapshot_usecase.py:1) 可独立从状态缓存读取完整快照，组装 `StateSnapshotMessage` 并通过注入的 publisher 发布

### 7.3 分层职责

#### framework

负责运行支撑和系统装配。

- `framework/persistence/`
  数据库 `Base`、`Session`、ORM、建库、样例数据导入

#### runtime

负责采集调度。

- `scheduler.py`
  任务注册、作业状态维护、停止控制
- `scheduler_factory.py`
  APScheduler 实例创建
- `scheduler_job.py`
  作业内存状态结构

#### ports

定义应用边界。

- `ports/runtime/`
  运行时任务配置读取接口
- `ports/source/`
  采集定义、采集 adapter、adapter registry 接口
- `ports/state/`
  本地状态缓存写入与完整快照读取接口
- `ports/message/`
  快照消息发布接口

#### adapters

负责边界实现。

- `adapters/config/`
  从数据库读取运行时配置和 OPC UA 采集定义
- `adapters/source/`
  OPC UA 采集实现、静态 registry
- `adapters/state/`
  SQLite 本地状态缓存、Redis 本地状态缓存
- `adapters/message/`
  RelationalOutbox、Redis Streams、Kafka 快照消息发布实现

#### usecases

负责应用层业务动作。

- `pull_source_state_usecase.py`
  一次性读取或单次轮询批量执行
- `subscribe_source_state_usecase.py`
  长运行订阅执行
- `emit_state_snapshot_usecase.py`
  读取完整状态快照、组装快照消息并发布
- `roles/`
  采集请求构造、状态更新、订阅协作、快照组装与发布等角色逻辑
- `dtos/`
  应用层传输对象与发布结果对象

### 7.4 配置模型与状态模型

当前数据库核心表：

- `substation` — 场站/电站注册
- `device` — 设备资产（关联场站，含设备编码、型号、线路号）
- `acquisition_model` — 采集模型标识（model_id + version 唯一）
- `acquisition_variable` — 模型下变量定义（variable_key + locator + locator_type）
- `acquisition_task` — 采集任务（绑定设备+模型，含协议、采集模式、连接参数、启用状态）
- `variable_state` — SQLite/关系型本地最新状态缓存
- `state_snapshot_outbox` — 关系型 outbox 快照消息表

Redis 状态缓存结构（生产推荐）：

- 以 `hash_key:station_id` 为维度组织 Redis Hash
- 每个 field 存储一个变量的完整快照行（JSON payload）
- 包含 `station_id`、`device_code`、`model_id`、`variable_key`、`value`、`source_observed_at`、`received_at`、`updated_at`

### 7.5 后端选择机制

`config.py` 通过三个环境变量独立选择三个后端维度：

| 维度 | 环境变量 | 可选值 | 默认值 |
| ---- | ------- | ----- | ----- |
| 数据库 | `WHALE_INGEST_DATABASE_BACKEND` | `sqlite` / `postgresql` | `postgresql` |
| 状态缓存 | `WHALE_INGEST_STATE_CACHE_BACKEND` | `relational` / `redis` | `redis` |
| 消息发布 | `WHALE_INGEST_MESSAGE_BACKEND` | `relational_outbox` / `redis_streams` / `kafka` | `kafka` |

限制：`relational` 状态缓存目前要求数据库后端为 `sqlite`。

## 9. 已实现功能

当前已经实现并通过测试验证的能力：

**基础设施**
- SQLite 数据库初始化与 PostgreSQL 数据库初始化
- 样例数据导入（3 变量基础版 + GB/T 30966.2 全字段版 ~530 字段）
- Docker Compose 一键开发环境（PostgreSQL + Redis + Kafka）
- 通过环境变量独立选择 database / state cache / message 三个后端

**运行时调度**
- `ONCE` 模式调度执行
- `POLLING` 模式调度执行
- `SUBSCRIPTION` 模式调度执行
- 同协议、同周期任务合并
- 统一停止控制与 `stop_event` 订阅停止

**数据采集**
- 基于数据库的运行时任务读取与 OPC UA 采集定义构造
- OPC UA 单次读取（`read`）
- OPC UA 订阅采集（`subscribe` + `datachange_notification`）
- 订阅启动前 initial read 全量初始化
- OPC UA 仿真器：正弦波 + 高斯噪声模拟，支持单机/多机，支持自定义变量值

**状态缓存**
- 采集结果写入 SQLite `variable_state` 本地状态缓存
- 采集结果写入 Redis 本地状态缓存（生产推荐）
- 从 SQLite / Redis 读取完整当前状态快照

**消息发布**
- `StateSnapshotMessage` 统一快照消息组装
- RelationalOutbox 快照消息发布
- Redis Streams 快照消息发布
- Kafka 快照消息发布

### 10.2 启动方法

**前置条件**：在项目根目录，Python 环境已激活。

```bash
# 1. 启动开发基础设施（PostgreSQL + Redis + Kafka）
docker compose -f docker-compose.ingest-dev.yaml up -d

# 2. 单元测试 + 冒烟测试（无需 Docker）
python -m pytest tests/unit/ tests/smoke/ -v

# 3. 集成测试（无需 Docker，使用 sqlite + 内存缓存）
python -m pytest tests/integration/ -m integration -v

# 4. E2E 测试（需要 Docker 基础设施运行中）
python -m pytest tests/e2e/ -m e2e -v

python -m pytest tests/e2e/test_instrumented_pipeline_report.py -v -s

# 6. 压力测试（需要 Docker 基础设施，运行时间较长）
python -m pytest tests/stress/ -m stress -v

docker compose -f docker-compose.ingest-dev.yaml down --volumes
```

**注意**：
- E2E 测试和压力测试依赖 Docker 容器中的 PostgreSQL（5432）、Redis（16379）、Kafka（9092）
- 集成测试使用 sqlite + 内存状态缓存，无需外部服务
- E2E conftest 会自动清理数据库残留数据，可重复运行

## 11. 未实现功能

以下能力目前尚未实现，或仍停留在抽象边界与局部测试支持层面：

- 非 OPC UA 协议接入
  例如 Modbus、IEC 61850、IEC 104、MQTT、文件导入等
- 将 `EmitStateSnapshotUseCase` 正式接入 scheduler 或独立运行入口，形成稳定的定时快照发射链路
- 历史数据归档与回放
  当前只维护最新状态，不维护历史时序表
- 运行时配置热更新
  当前通过 scheduler reload 机制可重装，但没有完整的动态配置管理流程
- 生产级订阅恢复与断线重连策略细化
  当前有基础停止与重新启动能力，但没有完整的健壮性治理策略
- 完整迁移机制
  当前以建库/重建为主，没有正式数据库 migration 体系
- 多租户、多项目隔离
- 采集结果质量码、健康状态、告警状态统一治理
- 与 `stream-ingestion / raw-store / batch-processing / serving` 的正式平台级集成闭环

## 12. 当前实现与旧方案的差异说明

旧版方案文档里包含了不少面向未来的设计，例如：

- `MaintainSourceStateUseCase`
- 多协议统一 source strategy 目录
- sink/publisher 分层
- 以内存状态缓存作为主热路径

这些设计方向可以作为架构备选思路参考，但它们不是当前代码的实际实现。

当前实现更聚焦，也更具体：

- 主入口是 scheduler
- 主 use case 是 `PullSourceStateUseCase` 与 `SubscribeSourceStateUseCase`
- 已实现 `EmitStateSnapshotUseCase`，用于读取完整本地状态快照并发布统一快照消息
- 当前唯一正式 source adapter 是 OPC UA
- 当前正式本地状态缓存实现包括 SQLite 与 Redis
- 测试中额外提供 `ModeCaptureStateCache` helper 便于观察输出
- 当前快照消息发布实现包括 RelationalOutbox、Redis Streams 与 Kafka
- 开发环境通过 Docker Compose 提供完整 PostgreSQL + Redis + Kafka 栈
- GB/T 30966.2-2022 字段模型已定义，支持 ~530 字段的风电机组全量数据点
- E2E 测试覆盖完整 PostgreSQL → Redis → Kafka/Redis Streams 链路
- 压力测试框架已就绪，支持多风机、多字段并发场景

因此，本说明书以”当前已落地代码”为准，而不再沿用旧方案中的目标架构描述。
