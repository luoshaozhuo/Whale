# Python 工程工具与代码文档规范说明

本文档用于说明 Python 项目中常用的工程质量工具及代码文档规范，包括 `mypy`、`ruff`、Google-style docstring、文件头说明、函数说明和关键逻辑注释。可根据项目实际情况裁剪后放入 `AGENTS.md`、`.codex/policies/python-style.md` 或团队编码规范中。

---

# 1. 总体目标

在 Python 工程项目中，仅依靠测试并不足以保证代码长期稳定。对于结构较复杂的项目，尤其是包含调度、异步、协议适配、性能测试、数据模型转换等逻辑的项目，应同时建立以下几类约束：

1. **类型约束**：通过 `mypy` 检查函数参数、返回值、接口边界和数据结构是否一致。
2. **代码质量约束**：通过 `ruff` 检查未使用变量、导入顺序、潜在错误、坏味道和部分格式问题。
3. **行为验证约束**：通过 `pytest` 验证代码功能是否符合预期。
4. **文档与注释约束**：通过模块级文件头、Google-style docstring 和关键逻辑注释，保证代码可读、可维护、可交接。
5. **格式化约束**：通过 `ruff format` 或 `black` 统一代码格式，减少无意义的格式差异。

一句话概括：

> `ruff` 管代码是否干净，`mypy` 管类型边界是否稳定，`pytest` 管行为是否正确，docstring 和注释管代码是否可理解、可维护。

# 2. mypy 的作用与应用

## 2.1 mypy 是什么

`mypy` 是 Python 的静态类型检查工具。它根据代码中的类型注解，在代码运行之前检查类型是否匹配。

例如：

```python
def add(a: int, b: int) -> int:
    return a + b

x: str = add(1, 2)
```

这段代码运行时不一定立即报错，但 `mypy` 会提示：`add()` 返回的是 `int`，却赋值给了 `str` 类型变量。

---

## 2.2 mypy 主要检查什么

`mypy` 主要用于发现以下问题：

1. 函数参数类型是否正确。
2. 函数返回值类型是否正确。
3. 变量赋值类型是否一致。
4. `Optional` / `None` 是否被正确处理。
5. 抽象类、接口、Protocol 是否被正确实现。
6. 泛型、集合、字典、元组等复杂数据结构的类型是否一致。
7. 数据模型、事件对象、配置对象之间的字段类型是否稳定。

---

## 2.3 mypy 适合解决的问题

在工程项目中，`mypy` 特别适合约束以下内容：

1. **Port / Adapter 边界**

   例如 Clean Architecture 项目中，port 定义了抽象接口，adapter 必须严格实现该接口。

2. **Use Case 输入输出**

   use case 的输入、输出、异常、返回对象应保持稳定，避免调用方误用。

3. **配置对象**

   环境变量解析后形成的配置对象，应有明确类型，避免一会儿是 `str`，一会儿是 `float`，一会儿是 `None`。

4. **性能指标对象**

   例如 `r95`、`t95`、`j95`、`miss_rate`、`achievement_ratio` 等字段，应保持统一类型，避免后续汇总和报告生成出错。

5. **事件对象**

   调度事件、读取事件、错误事件等对象应有明确字段类型，避免事件传递链路中出现不一致。

---

## 2.4 mypy 示例

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class ReadMetrics:
    ok_read: int
    expected_read: int
    miss_rate: float
    r95_ms: float


def calculate_miss_rate(ok_read: int, expected_read: int) -> float:
    if expected_read <= 0:
        raise ValueError("expected_read must be greater than zero")
    return (expected_read - ok_read) / expected_read


metrics = ReadMetrics(
    ok_read=90,
    expected_read=100,
    miss_rate=calculate_miss_rate(90, 100),
    r95_ms="12.5",
)
```

上面代码中，`r95_ms` 被声明为 `float`，但实际传入了字符串 `"12.5"`。`mypy` 可以提前发现这类问题。

---

## 2.5 mypy 推荐命令

```bash
mypy src tests
```

也可以只检查核心源码：

```bash
mypy src
```

如果项目中测试代码类型约束不够严格，可以先只对源码启用严格检查，再逐步覆盖测试代码。

---

## 2.6 mypy 在项目中的建议原则

1. 新增公共函数、类、dataclass、Protocol、配置对象时必须写类型注解。
2. 不要随意使用 `Any`。如果必须使用，应说明原因。
3. 对 `Optional` 值使用前必须显式判断。
4. 对外暴露的 port、adapter、use case 应保持类型稳定。
5. 复杂字典建议逐步替换为 dataclass、TypedDict 或 Pydantic model。
6. 性能指标、调度事件、协议数据对象不应使用松散的 `dict[str, Any]` 长期传递。

# 3. ruff 的作用与应用

## 3.1 ruff 是什么

`ruff` 是一个速度很快的 Python 代码检查和格式化工具。它可以替代或覆盖很多传统工具的功能，例如：

1. `flake8`
2. `pyflakes`
3. `pycodestyle`
4. `isort`
5. 部分 `pylint` 规则
6. 部分 `black` 格式化能力

---

## 3.2 ruff 主要检查什么

`ruff` 主要用于发现以下问题：

1. 未使用的 import。
2. 未使用的变量。
3. 导入顺序不规范。
4. 行长度问题。
5. 变量命名问题。
6. 裸 `except`。
7. 过宽异常捕获。
8. 重复或低质量代码模式。
9. 潜在 bug。
10. 不推荐写法。
11. 复杂度过高。
12. 测试代码中的不规范写法。

---

## 3.3 ruff 示例

```python
import os
import sys


def foo():
    x = 1
    return 2
```

`ruff` 可能会提示：

1. `os` 未使用。
2. `sys` 未使用。
3. `x` 被赋值但未使用。
4. 函数缺少返回类型注解，取决于规则配置。

---

## 3.4 ruff 对异常处理的检查

不推荐：

```python
try:
    do_something()
except:
    pass
```

推荐：

```python
try:
    do_something()
except RuntimeError as exc:
    logger.warning("Failed to execute operation: %s", exc)
```

如果确实需要捕获宽泛异常，也应该说明原因：

```python
try:
    read_from_source()
except Exception as exc:
    # Keep the profiling run alive after one failed read. The failure is recorded
    # as a sample so that the final report can expose error rate and miss rate.
    record_failed_sample(exc)
```

---

## 3.5 ruff 推荐命令

检查代码：

```bash
ruff check .
```

自动修复部分问题：

```bash
ruff check . --fix
```

格式化代码：

```bash
ruff format .
```

常见组合：

```bash
ruff check .
ruff format .
mypy src tests
pytest
```

---

## 3.6 ruff 在项目中的建议原则

1. 不允许残留未使用的 import 和变量。
2. 导入顺序必须稳定。
3. 不允许裸 `except`。
4. 不允许无解释地吞掉异常。
5. 长函数、复杂函数应拆分或加阶段性注释。
6. 测试代码中也应避免大量重复、魔法数字和不可解释的断言。
7. 自动修复前应查看 diff，避免工具误改影响语义。

---

# 4. mypy 与 ruff 的区别

| 工具 | 核心作用 | 主要关注点 | 典型问题 |
|---|---|---|---|
| `mypy` | 类型检查 | 类型边界是否稳定 | 参数类型不对、返回值不对、Optional 未处理 |
| `ruff` | 代码质量检查 | 代码是否干净、规范、有潜在 bug | 未使用变量、导入混乱、裸 except、复杂度高 |
| `pytest` | 行为测试 | 代码运行结果是否符合预期 | 功能错误、边界行为错误、集成失败 |
| `ruff format` / `black` | 格式化 | 代码格式是否统一 | 缩进、换行、空格、行宽 |

简化理解：

> `mypy` 检查“类型是否对”，`ruff` 检查“代码是否干净”，`pytest` 检查“行为是否对”。

# 5. Google-style docstring 的作用与应用

## 5.1 Google-style docstring 是什么

Google-style docstring 是 Python 中一种常用的文档字符串格式。它用于说明模块、类、函数或方法的用途、参数、返回值、异常和补充说明。

示例：

```python
def read_source_once(source_id: str, timeout_s: float) -> SourceSnapshot:
    """Read one snapshot from a source.

    Args:
        source_id: Logical source identifier.
        timeout_s: Maximum time to wait for the read operation.

    Returns:
        A normalized source snapshot containing point values and timestamps.

    Raises:
        TimeoutError: If the source does not respond before ``timeout_s``.
        SourceReadError: If the source returns invalid data.
    """
```

---

## 5.2 Google-style docstring 解决什么问题

Google-style docstring 主要解决以下问题：

1. 函数职责不清楚。
2. 参数含义不清楚。
3. 返回值结构不清楚。
4. 异常条件不清楚。
5. 调用约束不清楚。
6. 设计意图无法从代码本身看出。
7. 长代码维护成本高。

---

## 5.3 docstring 与普通注释的区别

| 类型 | 位置 | 作用 |
|---|---|---|
| 模块级 docstring | 文件开头 | 说明整个文件的职责、边界和设计取舍 |
| 类 docstring | class 下方 | 说明类的职责、状态、约束 |
| 函数 docstring | function 下方 | 说明函数输入、输出、异常和行为 |
| 行内注释 | 关键代码附近 | 解释复杂逻辑为什么这样写 |
| 阶段注释 | 长函数内部 | 划分逻辑阶段，提升可读性 |

docstring 主要说明“这个对象是什么、怎么用、有什么约束”。

行内注释主要说明“这里为什么这样写”。

# 6. 文件头说明规范

## 6.1 文件头的作用

对于非平凡 Python 文件，应在文件开头写模块级 docstring。文件头不只是列出函数名，而是说明：

1. 这个文件负责什么。
2. 这个文件不负责什么。
3. 主要数据流或控制流是什么。
4. 关键设计边界是什么。
5. 是否涉及兼容性约束。
6. 是否涉及性能假设。
7. 是否涉及环境变量。
8. 是否涉及调度、并发、协议、指标等关键概念。

---

## 6.2 文件头示例

```python
"""Polling scheduler used by source simulation performance tests.

This module provides a small polling scheduler for periodically reading one or
more logical data sources. It is intentionally independent from OPC UA, asyncua,
and open62541. Protocol-specific reading is injected through a callback so that
the scheduler can be tested without a real server.

Design goals:
    1. Keep scheduling behavior explicit and measurable.
    2. Support both relative and absolute scheduling modes.
    3. Expose timing metrics for diagnosing missed reads, jitter, and callback
       latency.
    4. Avoid hiding timing errors behind silent retries.

This module should not contain protocol-specific read logic, OPC UA NodeId
construction, or simulator fleet initialization.
"""
```

---

## 6.3 文件头应该包含的内容

推荐结构：

```python
"""Short one-line summary.

Longer explanation of the module responsibility.

Main concepts:
    1. Concept A.
    2. Concept B.
    3. Concept C.

Design boundaries:
    1. This module does ...
    2. This module does not ...

Compatibility notes:
    1. Explain legacy behavior or environment variables if needed.

Performance notes:
    1. Explain timing assumptions or measurement constraints if needed.
"""
```

---

## 6.4 文件头不应该写什么

不推荐：

```python
"""This file contains functions."""
```

不推荐：

```python
"""Utility functions."""
```

不推荐：

```python
"""Author: xxx
Date: xxx
"""
```

文件头应重点说明模块职责和设计边界，而不是写空泛说明。

# 7. 函数 docstring 规范

## 7.1 基本结构

推荐使用以下结构：

```python
def calculate_read_metrics(
    samples: Sequence[ReadSample],
    expected_read_count: int,
) -> ReadMetrics:
    """Calculate aggregate read metrics from raw reader samples.

    Args:
        samples: Successful and failed read samples collected during the test.
        expected_read_count: Expected number of read attempts, usually computed
            as ``server_count * target_hz * duration_s``.

    Returns:
        Aggregated read metrics, including successful read count, missed read
        count, achievement ratio, and latency percentiles.

    Raises:
        ValueError: If ``expected_read_count`` is less than or equal to zero.

    Notes:
        ``missed_read`` is derived from the expected read count and successful
        read count. It does not necessarily mean that the server failed to return
        data; it may also indicate that the client scheduler did not issue enough
        read attempts.
    """
```

---

## 7.2 Args

`Args` 用于说明参数含义，而不是简单重复参数名。

不推荐：

```python
def read_source(source_id: str) -> SourceSnapshot:
    """Read source.

    Args:
        source_id: source_id.
    """
```

推荐：

```python
def read_source(source_id: str) -> SourceSnapshot:
    """Read one normalized snapshot from a logical source.

    Args:
        source_id: Logical source identifier used to load connection settings
            and point definitions.
    """
```

---

## 7.3 Returns

`Returns` 应说明返回值的语义。

不推荐：

```python
Returns:
    ReadMetrics.
```

推荐：

```python
Returns:
    Aggregated metrics computed from all reader samples, including success
    count, miss count, achievement ratio, and latency percentiles.
```

---

## 7.4 Raises

函数可能主动抛出的异常，应写入 `Raises`。

```python
def parse_positive_float(value: str, name: str) -> float:
    """Parse a positive floating-point value from text.

    Args:
        value: Raw string value to parse.
        name: Human-readable configuration name used in error messages.

    Returns:
        Parsed positive floating-point value.

    Raises:
        ValueError: If ``value`` is not a valid positive number.
    """
```

---

## 7.5 Notes

`Notes` 用于说明补充约束、设计意图或容易误解的点。

```python
def calculate_client_period_ratio(periods_ms: Sequence[float], target_ms: float) -> float:
    """Calculate the ratio of client sampling periods within tolerance.

    Args:
        periods_ms: Adjacent client read intervals in milliseconds.
        target_ms: Expected sampling period in milliseconds.

    Returns:
        Ratio of periods that satisfy the configured tolerance.

    Notes:
        This metric describes whether the client issued reads at the expected
        cadence. It is independent from whether each issued read succeeded.
    """
```

---

## 7.6 Examples

只有当用法不直观时才写 `Examples`。

```python
def build_node_path(ied: str, ld: str, ln: str, do: str) -> str:
    """Build a logical point path from IEC-style hierarchy segments.

    Args:
        ied: IED name.
        ld: Logical device name.
        ln: Logical node name.
        do: Data object name.

    Returns:
        Dot-separated logical path.

    Examples:
        >>> build_node_path("IED1", "LD0", "MMXU1", "TotW")
        'IED1.LD0.MMXU1.TotW'
    """
```

# 8. 类 docstring 规范

## 8.1 类 docstring 的作用

类 docstring 应说明：

1. 类的职责。
2. 类表示的概念。
3. 关键属性含义。
4. 状态是否可变。
5. 并发或生命周期约束。
6. 与其他类的关系。

---

## 8.2 dataclass 示例

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class PollingJobSpec:
    """Configuration for one periodic polling job.

    Attributes:
        source_id: Logical source identifier.
        interval_s: Polling interval in seconds.
        timeout_s: Maximum duration allowed for one read attempt.
        enabled: Whether this job should be registered by the scheduler.

    Notes:
        This object only describes scheduling intent. It must not contain
        protocol-specific connection objects or live client instances.
    """

    source_id: str
    interval_s: float
    timeout_s: float
    enabled: bool = True
```

---

## 8.3 普通类示例

```python
class SourcePollingScheduler:
    """Schedule periodic reads for logical data sources.

    The scheduler owns timing and task lifecycle management. It does not know
    how to read from a concrete protocol. Actual source reading is injected
    through a callback.

    Notes:
        The scheduler uses monotonic time for all internal timing calculations.
        Wall-clock time should only be used at reporting boundaries.
    """
```

# 9. 关键逻辑注释规范

## 9.1 注释应解释“为什么”，而不是重复“做了什么”

不推荐：

```python
# Add one to count.
count += 1
```

推荐：

```python
# Count only successful server reads. Failed reads are kept in raw samples but
# are excluded from achievement-rate calculation.
ok_read += 1
```

---

## 9.2 调度逻辑注释示例

```python
# Use an absolute next deadline instead of sleeping a fixed interval after each
# read. This prevents callback latency from accumulating into long-term drift.
next_deadline += interval_s
```

---

## 9.3 性能指标注释示例

```python
# Keep client-side period statistics separate from server-side update period.
# A low cli_p means the reader did not sample at the requested cadence even if
# every issued read succeeded.
client_periods_ms.append(now_ms - previous_read_ms)
```

---

## 9.4 异常处理注释示例

```python
try:
    sample = await read_once()
except Exception as exc:
    # Do not fail the whole profiling run on a single read error. The error is
    # recorded as a failed sample so that miss rate and error rate remain visible
    # in the final report.
    record_failed_sample(exc)
```

---

## 9.5 兼容性注释示例

```python
# Preserve the legacy environment variable name for compatibility with existing
# shell scripts. New code should use SOURCE_SIM_LOAD_SOURCE_UPDATE_ENABLED.
update_enabled = read_bool_env(
    "SOURCE_SIM_LOAD_SOURCE_UPDATE_ENABLED",
    fallback_names=["SOURCE_SIM_OPEN62541_INTERNAL_UPDATE_ENABLED"],
)
```

---

## 9.6 边界转换注释示例

```python
# Convert monotonic timestamps to relative milliseconds only at the reporting
# boundary. Internal calculations stay in seconds to avoid repeated conversion
# noise.
relative_ms = (monotonic_s - start_monotonic_s) * 1000.0
```

---

# 10. 长函数注释规范

## 10.1 长函数优先拆分

如果一个函数过长，应优先考虑拆分为多个小函数。例如：

1. `build_reader_plans()`
2. `run_reader_tasks()`
3. `collect_raw_samples()`
4. `aggregate_metrics()`
5. `render_report()`

---

## 10.2 不适合拆分时使用阶段注释

如果拆分会破坏上下文连续性，可以用阶段注释：

```python
# Phase 1: Build reader plans.
reader_plans = build_reader_plans(config)

# Phase 2: Start readers and collect raw samples.
samples = await run_reader_tasks(reader_plans)

# Phase 3: Aggregate metrics.
metrics = aggregate_metrics(samples)

# Phase 4: Render the final report.
render_report(metrics)
```

---

## 10.3 长函数中的注释重点

长函数中的注释应重点说明：

1. 当前阶段的目的。
2. 为什么不能提前返回。
3. 为什么某些异常被记录而不是抛出。
4. 为什么某些指标分开统计。
5. 为什么某些逻辑必须保持顺序。
6. 为什么某些兼容分支不能删除。

---

# 11. 测试代码文档规范

## 11.1 测试文件头应说明什么

性能测试、集成测试、端到端测试文件必须有文件头说明：

1. 这个测试验证什么。
2. 测试范围是什么。
3. 不测试什么。
4. 关键模式是什么意思。
5. 关键指标如何解释。
6. 哪些断言是硬性正确性检查。
7. 哪些断言是诊断性阈值。
8. 运行前需要哪些环境变量。

---

## 11.2 性能测试文件头示例

```python
"""Multi-server OPC UA source simulation profiling test.

This test profiles the maximum sustainable client sampling cadence when reading
from multiple simulator servers.

The test supports two reader modes:
    1. coroutine: all readers run in the same Python event loop.
    2. process: each reader runs in a separate Python process.

The test supports two read schedules:
    1. burst: all readers issue reads at the same logical tick.
    2. stagger: readers are offset across the sampling interval.

Main metrics:
    ok_read:
        Number of successful server read attempts.
    expected_read:
        Expected read attempts, computed from server count, target frequency,
        and test duration.
    miss_rate:
        Ratio of expected reads that were not successfully completed.
    r95/r99:
        Percentiles of the protocol read call duration.
    t95/t99:
        Percentiles of the full reader tick duration.
    j95/j99:
        Percentiles of client sampling jitter.
    cli_p:
        Ratio of client sampling periods that satisfy the target cadence
        tolerance.

This test is diagnostic. A failure may indicate server limitations, client event
loop contention, protocol decoding overhead, scheduling jitter, or post-read
processing cost.
"""
```

---

## 11.3 测试函数 docstring 示例

```python
def test_multi_server_profile_reaches_target_hz() -> None:
    """Profile whether multi-server reading reaches the configured target rate.

    The test records both strict pass/fail metrics and diagnostic timing data.

# 14. 推荐写入 AGENTS.md 或 Codex Policy 的规范

下面内容可直接放入 `AGENTS.md`、`.codex/policies/python-style.md` 或类似文件中。

```markdown
## Python Code Quality Policy

Use `ruff`, `mypy`, and `pytest` together to maintain code quality.

### Required checks

Before submitting Python changes, run the relevant subset of:

```bash
ruff check .
ruff format .
mypy src tests
pytest
```

For focused changes, run the smallest meaningful subset first, then expand to
related unit, integration, or performance tests.

### mypy

Use type annotations for public functions, public classes, dataclasses,
configuration objects, event objects, and architecture boundary interfaces.

Avoid unnecessary `Any`. If `Any` is required because of external libraries,
dynamic data, or compatibility boundaries, keep it local and document the reason.

Handle `Optional` values explicitly before use.

### ruff

Keep imports clean and ordered. Remove unused imports, unused variables, and
temporary debugging code.

Do not use bare `except`. Do not silently swallow exceptions unless the code
records the failure and explains why the process should continue.

Use `ruff format` or the project-approved formatter to keep formatting stable.

### Documentation and comments

Each non-trivial Python file must start with a module-level docstring explaining:

1. The responsibility of the module.
2. The main concepts or data flow implemented in the file.
3. Important design boundaries, including what the module must not do.
4. Any compatibility constraints, environment variables, or performance
   assumptions that affect the implementation.

Use Google-style docstrings for public classes, public functions, test helper
functions, and non-trivial private functions.

A function docstring should include, when applicable:

1. A one-sentence summary.
2. Args.
3. Returns.
4. Raises.
5. Notes.
6. Examples, only when they clarify non-obvious usage.

Inline comments should explain why the code is written this way, not simply
repeat what the next line of code does.

Add comments for complex or non-obvious logic, especially:

1. Scheduling logic.
2. Concurrency logic.
3. Performance measurement logic.
4. Compatibility branches.
5. Error-handling decisions.
6. Data model transformations.
7. Logic where the reason is more important than the operation itself.

Avoid noisy comments that merely restate obvious code.

### Long functions

Avoid unnecessarily long functions. When a function contains several logical
phases, split it into smaller helpers where practical.

If splitting would make the code harder to follow, use section comments:

```python
# Phase 1: Build reader plans.
# Phase 2: Start readers and collect raw samples.
# Phase 3: Aggregate metrics.
# Phase 4: Render the final report.
```

### Test code documentation

Performance and integration tests must document:

1. What behavior the test validates.
2. What each mode means, for example coroutine/process or burst/stagger.
3. Which metrics are collected and how they should be interpreted.
4. Which assertions are strict correctness checks and which are diagnostic
   thresholds.

Do not leave large performance tests as unexplained procedural scripts.
```

# 16. 代码文档的判断标准

一份好的代码文档，不是注释越多越好，而是做到以下几点：

1. 读者能快速知道这个文件负责什么。
2. 读者能快速知道这个文件不负责什么。
3. 读者能理解函数参数和返回值的语义。
4. 读者能理解关键指标的含义。
5. 读者能理解复杂逻辑为什么这样写。
6. 读者能知道哪些代码是兼容性保留，不能随便删除。
7. 读者能知道哪些异常是故意记录而不是抛出。
8. 读者能根据 docstring 正确调用函数。
9. 读者能根据注释定位性能瓶颈。
10. 后续接手的人不需要反复追问原作者。

## 17.1 反例：空泛文件头

不推荐：

```

改进：

This module converts raw reader samples into stable metrics used by profiling
reports. It does not start simulator servers, create OPC UA clients, or schedule
reader tasks.

The aggregation logic keeps protocol read latency, full reader tick duration,
and client sampling jitter as separate metrics so that bottlenecks can be
diagnosed without relying on a single coarse duration value.
"""

## 17.2 反例：机械注释

不推荐：

```python
# Get current time.
now = time.monotonic()
```

改进：

```python
# Use monotonic time for scheduler calculations so that wall-clock adjustments
# do not create false jitter or negative durations.

## 17.3 反例：不清楚的异常处理

不推荐：

```python

改进：

```python
try:
    await reader.read()

    # as a sample so that error rate remains visible in the final report.
    samples.append(ReadSample.failed(error=repr(exc)))

改进：

# 18. 对复杂 Python 项目的建议

对于包含调度、异步、协议适配、性能测试的项目，建议特别关注以下几类代码：

1. **调度器代码**

   必须写清楚调度模式、时间基准、错过周期如何处理、是否允许漂移。

2. **异步代码**

   必须写清楚任务生命周期、取消逻辑、异常传播方式、并发限制方式。

3. **协议适配代码**

   必须写清楚协议边界、外部库对象是否允许传入核心层、错误如何转换。

4. **性能测试代码**

   必须写清楚测试目标、指标定义、采样方法、断言含义。

5. **数据模型代码**

   必须写清楚字段语义、单位、是否可选、是否是运行时派生字段。

6. **环境变量解析代码**

   必须写清楚默认值、兼容变量、优先级和错误处理策略。
