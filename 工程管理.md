# 工程管理方法论：低成本自动化驱动的演化型增量迭代开发模型

## 1. 模型名称

本项目采用：

**低成本自动化驱动、项目记忆增强的演化型增量迭代开发模型**

该模型结合以下机制：

1. **目标约束稳定**
2. **需求演化细化**
3. **技术探查先行**
4. **证据驱动决策**
5. **增量迭代交付**
6. **代码质量门禁**
7. **测试闭环保护**
8. **低成本自动化优先**
9. **项目记忆沉淀**
10. **按需重构**
11. **文档治理与精简**

本模型适用于目标明确但实现路径、性能瓶颈、技术选型和需求细节需要持续探索的工程项目，尤其适用于由 AI 编码代理参与主要实现的项目。

---

## 2. 采用该模型的原因

本项目不是复现一个已经完全确定的软件，而是在明确工程目标和部署约束下探索可行实现方案。

项目的长期目标相对稳定，例如：

1. 支持高频数据采集。
2. 支持多设备并发接入。
3. 能够部署在普通服务器或边缘设备上。
4. 满足性能、稳定性、可诊断性和可维护性要求。
5. 保持代码可读、可扩展、可测试、可调试。

但具体实现路径具有不确定性，例如：

1. 现有协议库是否满足性能要求。
2. 客户端、服务端、调度器、数据解码或后处理哪个环节会成为瓶颈。
3. 单进程、异步、多进程或其他并发模型哪种更合适。
4. 性能指标和测试方法是否足够准确。
5. 现有代码结构是否能支撑下一阶段演进。
6. 哪些工作应由 AI 编码代理完成，哪些工作应由脚本、测试、工具或 skill 自动完成。

因此，本项目不适合一次性固定完整需求、完成全部设计后再编码的纯瀑布模式，也不适合无边界、无基线、无自动化、无质量门禁的随意敏捷模式。

更合理的方式是：

> 在稳定目标和当前基线之上，通过技术探查、最小验证和测试证据逐步细化需求；通过小步实现、小步验证和质量门禁控制风险；通过脚本、测试和 skill 降低重复执行成本；通过项目记忆和文档治理保证经验可继承、文档不膨胀；当代码、测试或文档结构阻碍继续演化时，按需重构。

## 3. 模型核心思想

### 3.1 目标稳定，路径演化

项目首先明确长期目标和关键约束，但不假设所有实现方案一开始就能确定。

需求、设计和计划会随着以下证据不断演化：

1. 技术调研结果。
2. 最小技术验证结果。
3. 性能测试结果。
4. 运行日志和 profile 数据。
5. 代码实现中的边界暴露。
6. 测试失败和调试结果。
7. 部署环境约束。
8. 工具链和自动化能力变化。

演化不是随意变化，而是基于证据的逐步收敛。

---

### 3.2 复用优先，自研克制

在进入自研之前，先判断是否存在可复用的成熟方案。

基本顺序为：

```text
现有方案调研
  ↓
候选方案筛选
  ↓
最小技术验证
  ↓
差距分析
  ↓
路线决策
```

路线决策包括：

1. 直接复用。
2. 封装复用。
3. 局部替换。
4. 深度改造。
5. 自研最小必要部分。

原则是：

> 能复用不自研，能封装不重写，能局部替换不整体推倒，只有关键约束无法满足时才自研。

---

### 3.3 证据驱动，而不是猜测驱动

每一轮计划应来自真实证据，而不是主观猜测。

常见证据包括：

1. 测试结果。
2. 性能指标。
3. profile 数据。
4. 日志。
5. 代码 review 结论。
6. 实验对照结果。
7. 外部库能力验证。
8. 质量检查结果。
9. 自动化脚本输出。

每一轮都应回答：

```text
当前证据说明了什么？
下一步最小可验证假设是什么？
做什么改动可以验证这个假设？
验证结果将如何影响下一步决策？
```

---

### 3.4 小步增量，持续可运行

每一轮开发只推进一个或少数几个可验证的小目标。

每个增量都应满足：

1. 可运行。
2. 可测试。
3. 可 review。
4. 可回退。
5. 可复盘。
6. 可作为下一轮基线。

不应长期保留“半成品状态”。

---

## 4. 代码质量是过程模型的一部分

代码质量不是编码完成后的附加要求，而是每轮迭代的准入和退出条件。

### 4.1 代码质量目标

代码应满足：

1. **可读性**：命名准确，结构清晰，职责明确。
2. **可维护性**：修改局部功能不应牵连大量无关模块。
3. **可扩展性**：新协议、新后端、新指标、新测试模式应能在清晰边界内加入。
4. **可测试性**：核心逻辑可通过单元测试、集成测试或性能测试验证。
5. **可调试性**：日志、指标、异常和报告能够帮助定位问题。
6. **边界清晰**：核心逻辑、协议适配、调度、指标统计、报告输出不应混杂。
7. **类型稳定**：关键数据结构、接口、配置对象和指标对象应保持明确类型。
8. **注释适度**：复杂逻辑、性能指标、兼容分支和设计取舍应有必要说明。

---

### 4.2 代码质量门禁

每轮实现后，应根据影响范围运行必要检查：

```bash
python -m py_compile <changed-python-files>
ruff check <changed-python-files>
mypy <related-package-or-files>
pytest <related-tests> -q
```

涉及全局改动时：

```bash
ruff check .
mypy src tests
pytest
```

涉及性能测试时：

```bash
pytest <performance-test-file> -s -v
```

质量门禁不是形式检查，而是防止以下问题进入基线：

1. 导入错误。
2. 类型边界破坏。
3. 未使用代码和临时调试残留。
4. 测试缺失。
5. 指标命名混乱。
6. 异常处理被吞掉。
7. 兼容性被无意破坏。
8. 性能问题被错误隐藏。

---

### 4.3 不允许用“改弱标准”代替修复问题

以下做法不应作为正常修复手段：

1. 为了通过测试而降低断言。
2. 为了避免报错而删除测试。
3. 为了绕开类型错误而滥用 `Any`。
4. 为了减少失败而吞掉异常。
5. 为了让性能测试通过而修改指标含义。
6. 为了减少 Codex 工作量而保留明显混乱结构。

如果确实需要调整标准，必须说明：

1. 原标准为什么不合理。
2. 新标准依据是什么。
3. 是否影响历史对比。
4. 是否需要记录到决策或实验文档中。

---

## 5. 测试闭环是过程模型的核心

测试不是最后阶段，也不是只为“证明能跑”。测试应贯穿需求细化、设计、编码、验证和重构。

### 5.1 测试的作用

测试承担以下职责：

1. 验证功能是否满足需求。
2. 固化历史 bug，防止回归。
3. 保护重构。
4. 验证架构边界。
5. 验证性能假设。
6. 验证配置兼容。
7. 暴露瓶颈和不稳定因素。
8. 作为项目记忆的一部分。

---

### 5.2 测试分层

建议采用分层测试策略：

```text
unit          单元测试，验证纯逻辑和小模块
integration   集成测试，验证多模块协作
smoke         冒烟测试，验证主流程可启动、可跑通
e2e           端到端测试，验证完整链路
performance   性能测试，验证吞吐、延迟、抖动和瓶颈
regression    回归测试，固化已修复问题
```

测试目录组织原则：

1. 正式源码测试集中放在根目录 `tests/`。
2. 独立工具测试可以放在 `tools/<tool_name>/tests/`。
3. 不建议把测试文件混放到 `src/` 源码目录。
4. 不建议所有测试平铺在一个目录下。
5. 性能测试应独立隔离，避免被普通快速测试误跑。

---

### 5.3 每轮都要明确测试策略

每一轮开始前应明确：

1. 是否需要新增测试。
2. 是否需要修改测试。
3. 是否需要删除过时测试。
4. 是否需要新增回归测试。
5. 是否需要运行性能测试。
6. 哪些测试是本轮验收条件。
7. 哪些测试只是诊断性输出。

修 bug 时，原则上应先补回归测试，再修实现。

---

### 5.4 性能测试必须可诊断

性能测试不能只输出 pass / fail，也不能只输出大量调用栈。

性能测试应尽量拆分链路指标，例如：

```text
scheduler wake lag
client read call duration
decode duration
post-process duration
reader tick duration
client period jitter
server update period
per-server success/miss/error
```

性能测试失败时，应能帮助判断问题更可能出在：

1. 调度器。
2. event loop。
3. client 库。
4. server 后端。
5. 协议解码。
6. 数据后处理。
7. 进程模型。
8. 测试参数。
9. 部署环境。

---

## 6. 低成本自动化优先

由于项目主要由 AI 编码代理参与实现，必须控制执行成本。不能把所有重复工作都交给 Codex 通过自然语言反复执行。

原则是：

> 能用脚本自动完成的，不靠 Codex 手工重复；能用测试验证的，不靠 Codex 口头判断；能用 skill 固化的，不靠临时提示词；能用规则检查的，不靠人工提醒。

---

### 6.1 自动化优先级

推荐按以下优先级处理重复性工作：

```text
1. 现有工具 / 标准命令
2. 项目脚本
3. 测试用例
4. pre-commit / CI
5. Codex skill
6. Codex 临时执行
7. 人工检查
```

优先使用低成本、可重复、可审计的方式。

---

### 6.2 适合脚本化的工作

以下工作不应长期依赖 Codex 每次手动完成：

1. 运行固定测试组合。
2. 生成测试报告。
3. 汇总性能结果。
4. 检查目录结构。
5. 检查禁止导入。
6. 检查废弃文件是否被恢复。
7. 检查环境变量命名。
8. 检查指标字段命名。
9. 检查 Markdown 文档结构。
10. 生成 current-state 草稿。
11. 生成 experiment 记录模板。
12. 生成 ADR 模板。

这些应逐步沉淀为：

```text
scripts/
tools/
Makefile
justfile
pytest marker
pre-commit hook
CI job
Codex skill
```

---

### 6.3 Codex 的合理使用边界

Codex 更适合处理：

1. 阅读和理解复杂代码。
2. 提出实现方案。
3. 修改代码。
4. 补测试。
5. 分析失败原因。
6. 总结变更。
7. 提炼经验。
8. 生成规则草案。
9. 设计重构方案。

Codex 不适合长期反复承担：

1. 每天重复运行同一组命令。
2. 每次手工检查同一类目录问题。
3. 每次手工汇总固定格式报告。
4. 每次靠自然语言记住禁止事项。
5. 每次手工判断显然可规则化的质量问题。

这类事项应自动化。

---

### 6.4 skill 的使用原则

当某类任务反复出现，且具有稳定输入、稳定步骤、稳定输出时，应考虑抽象为 skill。

适合 skill 化的任务包括：

1. 性能测试结果汇总。
2. current-state 更新。
3. Codex prompt 生成。
4. 失败复盘模板生成。
5. ADR 生成。
6. 实验记录生成。
7. 测试目录检查。
8. 项目基线检查。
9. 代码质量检查清单生成。

原则是：

> 临时任务用 prompt；重复任务用脚本；复杂但模式稳定的任务用 skill；关键质量约束用测试或 CI。

---

## 7. 项目记忆增强

项目记忆不是无限堆文档，而是把关键经验转化为可读取、可执行、可验证的资产。

### 7.1 项目记忆类型

项目记忆包括：

1. 工程规则。
2. 架构决策。
3. 历史错误。
4. 实验记录。
5. 测试用例。
6. 当前基线。
7. 性能结果。
8. 已知限制。
9. 自动化脚本。
10. 质量门禁。

其中，最可靠的项目记忆不是长篇说明，而是：

1. 测试。
2. 脚本。
3. 规则检查。
4. ADR。
5. 简洁基线文档。

---

### 7.2 经验沉淀规则

建议采用：

```text
一次错误 → lesson
重复错误 → policy
架构取舍 → ADR
实验结果 → experiment record
性能结果 → performance record
阶段状态 → current-state
重复操作 → script / skill
关键行为 → test
```

这能避免所有经验都堆到一个巨大文档里。

---

### 7.3 项目记忆不能只依赖 Codex

不应期待 Codex 仅凭上下文长期记住所有经验。

经验应沉淀到仓库中，并尽量转化为：

1. 可执行命令。
2. 可失败的测试。
3. 可检查的规则。
4. 可引用的决策记录。
5. 可更新的当前基线。

---

## 8. 文档治理：文档也是重构对象

文档不能无限膨胀。文档越多，维护成本越高；文档过旧，反而会误导 Codex 和人工判断。

因此，文档本身也需要治理和重构。

---

### 8.1 文档的作用边界

文档应解决以下问题：

1. 说明目标。
2. 记录当前状态。
3. 记录重要决策。
4. 记录实验结论。
5. 记录无法由测试表达的设计理由。
6. 指导后续任务。
7. 降低交接成本。

文档不应承担以下职责：

1. 重复代码能表达的细节。
2. 重复测试能验证的行为。
3. 重复脚本能生成的报告。
4. 长期保存过时过程草稿。
5. 无限堆叠临时结论。
6. 替代自动化检查。

---

### 8.2 文档精简原则

文档应遵循：

1. **少而准**：只保留对后续决策有价值的信息。
2. **短而硬**：规则要明确，避免长篇空泛说明。
3. **可更新**：状态类文档必须能被持续维护。
4. **可替代**：能用测试、脚本、ADR 表达的，不堆到总文档里。
5. **可归档**：过时实验和历史草稿应归档，不影响当前判断。
6. **可重构**：当文档结构阻碍查找和使用时，应重构文档体系。

---

### 8.3 文档重构触发条件

出现以下情况时，应重构文档：

1. 同一规则出现在多个文件且表述不一致。
2. current-state 过长，难以判断当前真实状态。
3. ADR 与现有实现矛盾。
4. 实验记录没有结论或无法支持决策。
5. Codex 经常引用过时文档。
6. 文档中的历史背景淹没当前规则。
7. 新任务开始前需要读太多文档才能行动。
8. 文档已经不能指导测试、实现或决策。

---

### 8.4 文档治理策略

建议将文档分为四类：

```text
1. 当前有效文档
2. 决策记录文档
3. 实验和性能记录文档
4. 历史归档文档
```

推荐结构：

```text
docs/
  current-state.md
  decisions/
  experiments/
  performance/
  archive/
```

规则：

1. `current-state.md` 必须短，记录当前状态，不写完整历史。
2. `decisions/` 记录重要取舍，不频繁修改历史 ADR，只追加新 ADR 或更新状态。
3. `experiments/` 记录实验假设、参数、结果和结论。
4. `performance/` 记录阶段性性能基线和对比。
5. `archive/` 存放过时草稿、旧方案和历史记录。
6. 过时内容不应继续留在当前工作入口中干扰判断。

---

## 9. 按需重构

按需重构不仅包括代码，也包括测试、脚本、文档和项目记忆。

### 9.1 重构对象

可被重构的对象包括：

1. 生产代码。
2. 测试代码。
3. 性能测试脚本。
4. 指标体系。
5. 工程脚本。
6. 文档结构。
7. Codex policies。
8. lessons。
9. current-state。
10. 项目目录结构。

---

### 9.2 重构触发条件

当以下问题开始阻碍下一步演化时，应考虑重构：

1. 新增功能越来越困难。
2. 测试越来越难写。
3. 调试路径越来越长。
4. 性能指标越来越难解释。
5. 同类逻辑重复出现。
6. 文档越来越难读。
7. 自动化命令越来越分散。
8. Codex 反复误改同类位置。
9. 一次改动牵连大量无关文件。
10. 旧结构已经无法支撑下一阶段目标。

---

### 9.3 重构原则

重构应遵循：

1. 行为不变。
2. 测试保护。
3. 小步修改。
4. 可回退。
5. 不顺手改变无关功能。
6. 重构前明确目的。
7. 重构后更新必要文档和测试。
8. 不为“好看”重构，只为降低后续成本重构。

---

## 10. 基本过程循环

本模型的基本循环为：

```text
项目记忆读取
  ↓
当前基线确认
  ↓
目标约束确认
  ↓
技术探查 / 现有方案调研
  ↓
最小技术验证
  ↓
证据分析
  ↓
需求细化
  ↓
当轮小设计
  ↓
小步实现
  ↓
代码质量检查
  ↓
测试验证
  ↓
自动化沉淀判断
  ↓
复盘与经验沉淀
  ↓
按需重构判断
  ↓
文档治理判断
  ↓
基线更新
  ↓
进入下一轮
```

简化表达为：

```text
记忆 → 基线 → 探查 → 证据 → 细化 → 实现 → 质量 → 测试 → 自动化 → 复盘 → 重构 → 新基线
```

## 11. 每轮迭代的最小产出

每一轮迭代至少应产生以下结果之一：

1. 一个可运行的代码增量。
2. 一个通过测试保护的缺陷修复。
3. 一个新的测试或回归测试。
4. 一个明确的实验结论。
5. 一个更准确的性能判断。
6. 一个必要的重构结果。
7. 一个自动化脚本或 skill。
8. 一个被记录的架构决策。
9. 一个更新后的当前基线。
10. 一个文档精简或归档结果。

如果一轮工作没有形成任何可继承结果，说明该轮过程需要复盘。

---

## 12. 每轮计划应包含的要素

每一轮开始前，应明确：

1. 当前基线是什么。
2. 当前主要问题是什么。
3. 本轮目标是什么。
4. 本轮不做什么。
5. 本轮依据的证据是什么。
6. 本轮要验证的假设是什么。
7. 允许修改的范围是什么。
8. 需要运行哪些代码质量检查。
9. 需要运行哪些测试。
10. 是否需要新增或修改自动化。
11. 是否涉及文档更新或文档精简。
12. 验收标准是什么。
13. 失败后如何判断下一步。

---

## 13. 每轮结束应回答的问题

每一轮结束后，应回答：

1. 本轮完成了什么。
2. 哪些目标没有完成。
3. 代码质量检查结果如何。
4. 测试结果如何。
5. 性能结果如何。
6. 是否发现新的问题。
7. 是否需要调整下一轮需求。
8. 是否需要补充测试。
9. 是否需要脚本化或 skill 化。
10. 是否需要重构代码、测试或文档。
11. 是否需要记录新的规则、错误、实验或架构决策。
12. 当前是否可以形成新的稳定基线。

## 15. 与传统过程模型的关系

本模型不是完全否定瀑布，也不是简单套用敏捷。

它吸收了瀑布模型中“需求、设计、编码、测试、验收”的阶段意识，但将其缩小到每一轮小迭代内部。

它吸收了敏捷模型中“持续反馈、快速调整、小步交付”的思想，但更强调：

1. 技术探查。
2. 性能证据。
3. 代码质量门禁。
4. 测试闭环。
5. 自动化优先。
6. 项目记忆。
7. 文档治理。
8. 按需重构。

因此，本模型可以理解为：

> 以演化型增量迭代为主体，以技术探查和证据分析驱动需求细化，以代码质量和测试闭环保证工程可靠性，以脚本、工具和 skill 降低重复成本，以项目记忆保证经验继承，以按需重构控制长期复杂度。

---

## 16. 一句话定义

**低成本自动化驱动、项目记忆增强的演化型增量迭代开发模型**，是指在稳定工程目标和当前基线之上，通过技术探查、最小验证和测试证据不断细化需求，小步实现和验证可运行增量，通过代码质量门禁和测试闭环控制风险，通过脚本、工具和 skill 降低重复执行成本，并将规则、错误、决策、实验和测试结果沉淀为项目记忆，在代码、测试、脚本或文档结构阻碍继续演化时按需重构的一种工程管理方法。
