# Python 代码风格与模式选型规则

## 基本代码风格

- Python 代码整体遵循项目约定与 PEP 8。
- 代码格式以仓库中的 `pyproject.toml` 配置为准。
- Black 为格式化基准，行宽以项目配置为准。
- 导入顺序与分组以 isort 结果为准。
- Flake8 规则以仓库中的 `.flake8` 配置为准。
- 类型检查以 mypy 配置与检查结果为准。
- 不要保留未使用的 import、变量、参数和调试残留。
- 优先编写清晰、直接、低嵌套的代码。
- 若逻辑明显分段，应优先提取为职责明确的辅助函数。
- 不要引入与当前任务无关的格式化噪音。

## 实现优先级总原则

- 简单场景优先普通函数和普通类组合，不为小问题引入复杂模式。
- 默认优先组合，不为少量代码复用引入继承层级。
- 只有在存在明确稳定的 is-a 关系时，才使用继承。
- 模式的目标是降低复杂度、提升可读性与可维护性，不是展示技巧。
- 当某种模式让调用链、职责边界或行为顺序变得更难理解时，应退回更直接的实现方式。

## Python 模式选型约束

### 组合与继承

- 默认优先组合，而不是继承。
- 当对象之间主要是“协作关系”而不是“类型从属关系”时，使用组合。
- 当继承开始依赖隐式调用顺序、跨层行为覆盖或脆弱基类假设时，应退回组合。
- 不要为了复用少量方法而建立多层继承结构。

### 装饰器模式

- 函数或方法级横切增强优先使用装饰器。
- 装饰器适用于日志、缓存、权限校验、重试、性能统计等横切场景。
- 装饰器不应用于隐藏核心业务语义。
- 若多层装饰器使主流程难以理解，应改为显式流程编排或普通函数封装。
- 装饰器应保持单一职责，避免一个装饰器同时承担多种行为。

### Mixin 与 MRO

- 类级横切能力复用优先使用 mixin。
- mixin 适用于审计、日志、权限、生命周期 hook、通用能力注入等横切场景。
- mixin 不应用于承载主业务流程。
- 每个 mixin 只负责单一能力，避免混合多个职责。
- 依赖 MRO 的调用链必须可解释、可预测，避免隐式副作用。
- 当继承层级变深、super 调用链不清晰或行为顺序难以推断时，应退回组合或显式调用。

### 策略模式

- 当同一职责存在多种可替换实现时，优先使用策略模式，而不是堆叠 `if/elif`。
- 策略模式适用于多种计算规则、校验规则、解析规则、转换规则或路由规则。
- 策略对象应围绕单一变化点设计。
- 若不同分支仅有极小差异，不要强行抽成策略。

### 工厂模式

- 当对象创建依赖配置、类型标识、运行上下文或外部输入时，优先使用工厂封装创建逻辑。
- 简单场景优先简单工厂，不提前引入抽象工厂。
- 工厂负责对象创建与装配，不承载业务决策。
- 不要为了创建单一固定对象而引入工厂。

### 适配器模式

- 当外部接口、第三方库、协议对象与内部模型不一致时，优先使用适配器隔离差异。
- 协议解析、字段映射、兼容转换应尽量收敛在适配层。
- 不要将外部数据格式、第三方对象结构或协议细节直接渗透到核心业务对象。
- 适配器负责转换，不负责主业务决策。

### 模板方法

- 当流程主骨架稳定、仅少量步骤可变时，可使用模板方法。
- 若流程本身也频繁变化，优先使用组合或显式编排，不要硬套模板方法。
- 模板方法应保持骨架简明，避免把大量细节堆进父类。

## 数据建模约束

### 贫血模型与充血模型

- DTO、配置对象、序列化对象、纯数据载体优先采用贫血模型。
- 具备业务不变量、状态约束、领域行为的对象优先采用充血模型。
- 不要将领域核心对象退化为只有字段和简单访问器的贫血模型。
- 不要将纯传输对象过度设计为充血模型。

### DTO / VO / Entity 区分

- DTO 用于传输与边界交换，不承载业务行为。
- VO 用于表达不可变业务概念，优先保持不可变。
- Entity 用于承载身份、状态与业务行为。
- 不要混用 DTO、VO 和 Entity 的职责。
- 不要把临时字典长期替代明确的数据对象。

## 持久化与边界访问

### 仓储模式

- 持久化访问应通过仓储接口或清晰封装隔离，不让业务逻辑直接依赖底层存储细节。
- 仓储负责聚合读写，不承载业务决策。
- 简单脚本、一次性逻辑或极小范围原型不强制引入仓储。
- 当数据访问逻辑开始在多个地方重复出现时，应收敛为仓储或统一访问层。

### 门面模式

- 当一个用例需要协调多个子模块时，可使用门面提供稳定入口。
- 门面用于简化调用，不承载细碎业务规则。
- 若门面持续膨胀，应回到明确的应用服务或编排层拆分职责。

## 禁止事项

- 不为小问题引入复杂模式。
- 不同时叠加多种模式而没有明确收益。
- 不为追求“优雅”而增加调用链隐式性。
- 不把模式本身当作架构目标。
- 不为了通过检查或满足形式规范而牺牲代码可读性与行为清晰度。
