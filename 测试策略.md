# Python 项目测试规范与目录组织建议（Codex 使用版）

本文档用于指导 Codex 在 Python 项目中新增、修改、删除和运行测试。内容覆盖测试总体原则、测试分层、测试目录组织、pytest 使用规范、fixture 与 mock 原则、异步与调度测试、性能测试、回归测试以及 Codex 修改代码后的验证流程。

本文档适合放入：

```text
.codex/policies/testing-policy.md
```

也可以裁剪后合并进：

```text
AGENTS.md
.codex/policies/engineering-general.md
```

# 1. 测试的核心目标

测试不是为了堆数量，而是为了保证代码修改后仍然满足设计边界、业务行为和工程质量要求。

测试应服务于以下目标：

1. 验证模块职责是否成立。
2. 验证架构边界是否稳定。
3. 验证接口输入输出是否符合预期。
4. 验证异常路径是否被正确处理。
5. 验证历史 bug 不会回归。
6. 验证性能测试能够定位瓶颈，而不是只输出 pass / fail。
7. 帮助 Codex 在修改代码后选择最小但有效的验证范围。

一句话概括：

> 测试应当让代码修改可验证、问题定位可追踪、历史行为可回归。

---

# 2. Codex 修改代码时的测试原则

Codex 每次修改代码后，必须回答三个问题：

1. 本次改动影响了什么行为？
2. 现有测试是否覆盖了这些行为？
3. 是否需要新增、修改或删除测试？

如果实现逻辑发生变化，但测试没有任何变化，必须说明原因。

# 3. 测试目录组织原则

## 3.1 总体建议

建议采用**集中式测试目录**：

```text
tests/
```

不建议在 `src/` 每个源码目录中放置测试文件。

推荐：

```text
project/
  src/
    package/
      module_a/
      module_b/

  tests/
    unit/
    integration/
    smoke/
    e2e/
    performance/
```

不推荐：

```text
project/
  src/
    package/
      module_a/
        module_a.py
        test_module_a.py
```

---

## 3.2 为什么不建议把测试放进 src

不建议在 `src/` 目录中混放测试文件，原因包括：

1. 生产代码和测试代码边界不清楚。
2. 打包发布时容易误带测试文件。
3. Codex 修改代码时容易混淆实现文件和测试文件。
4. 大型项目后期不好管理。
5. 性能测试、集成测试和单元测试难以分层。
6. 不利于按测试类型选择最小验证范围。

---

## 3.3 为什么不建议 tests 下全部平铺

不推荐：

```text
tests/
  test_polling.py
  test_config.py
  test_backend_factory.py
  test_registry.py
  test_multi_server_profile.py
  test_scenario1.py
```

问题是：

1. 看不出测试类型。
2. 看不出测试对应哪个源码模块。
3. 单元测试、集成测试、性能测试混在一起。
4. Codex 不容易选择最小测试范围。
5. 性能测试可能被普通全量测试误跑。
6. 测试文件数量变多后难以维护。

---

## 3.4 推荐目录结构

推荐采用两级组织方式：

1. 第一层按测试类型划分。
2. 第二层按源码模块路径组织。

示例：

```text
tests/
  unit/
    shared/
      source/
        scheduling/
          test_polling.py
          test_metrics.py
        opcua/
          test_backend_factory.py

  integration/
    shared/
      source/
        scheduling/
          test_polling_scheduler.py
        opcua/
          test_asyncua_backend.py
          test_open62541_backend.py

  smoke/
    test_source_simulation_smoke.py
    test_scenario1_smoke.py

  e2e/
    test_scenario1_pipeline.py

  performance/
    source_simulation/
      test_multi_server_profile.py
```

这样组织后，Codex 可以根据改动范围选择合适测试：

```bash
pytest tests/unit/shared/source/scheduling -q
pytest tests/integration/shared/source/scheduling -q
pytest tests/performance/source_simulation/test_multi_server_profile.py -s -v
```

# 4. 对 tools 目录的特殊建议

如果 `tools/<tool_name>` 是相对独立的工具，可以保留工具内部测试目录。

例如：

```text
tools/
  source_simulation/
    adapters/
    domain.py
    registry.py
    tests/
      unit/
      integration/
      performance/
```

这种结构适合以下情况：

1. 工具相对独立。
2. 工具有自己的入口、配置和测试命令。
3. 工具测试和主项目测试边界明显。
4. 工具包含性能测试或外部协议测试。
5. 工具可能独立运行或单独演进。

因此，对于包含 `tools/source_lab` 的项目，可以采用并存结构：

```text
project/
  tests/
    unit/
    integration/
    smoke/
    e2e/
    performance/

  tools/
    source_simulation/
      tests/
        unit/
        integration/
        performance/
```

原则是：

1. `src/` 下的正式源码测试放在根目录 `tests/`。
2. 独立工具自身测试可以放在 `tools/<tool_name>/tests/`。
3. 不要把测试文件直接塞进 `src/` 源码目录。
4. 不要把所有测试平铺在一个 `tests/` 目录中。

# 5. 测试分层策略

## 5.1 unit：单元测试

单元测试用于验证纯逻辑或小范围模块行为。

适合覆盖：

1. 配置解析。
2. dataclass / value object。
3. 指标计算。
4. 时间窗口计算。
5. 路径拼接。
6. 错误转换。
7. 调度策略中的纯计算逻辑。
8. 数据清洗、归一化、映射规则。
9. 环境变量解析。

单元测试不应依赖：

1. 网络。
2. 真实数据库。
3. 真实 OPC UA server。
4. 文件系统中的不稳定路径。
5. 当前机器性能。
6. 当前真实时间。
7. 测试执行顺序。

---

## 5.2 integration：集成测试

集成测试用于验证多个模块之间的协作。

适合覆盖：

1. use case + repository adapter。
2. scheduler + fake reader。
3. config parser + runtime builder。
4. adapter + local test server。
5. 数据从输入到中间层的转换链路。
6. 本地 SQLite 或临时文件存储。

集成测试可以使用：

1. 临时目录。
2. 临时 SQLite。
3. fake server。
4. 本地 loopback 网络。
5. 小规模真实 adapter。

应避免：

1. 长时间运行。
2. 依赖外部网络。
3. 依赖生产配置。
4. 依赖固定端口。
5. 依赖测试执行顺序。

---

## 5.3 smoke：冒烟测试

冒烟测试用于验证主流程能否跑通。

适合覆盖：

1. CLI 能启动。
2. 关键入口能加载配置。
3. simulator 能创建。
4. 简单读写链路能走完。
5. 最小闭环能执行。
6. 关键 import 和依赖装配无异常。

冒烟测试不应覆盖过多细节。

---

## 5.4 e2e：端到端测试

端到端测试用于验证接近真实场景的完整链路。

适合覆盖：

1. 从配置加载到数据输出的完整链路。
2. simulator server + client reader + aggregation。
3. 数据库初始化 + 数据写入 + 查询。
4. 多组件协同。
5. scenario 级业务闭环。

端到端测试应数量少、场景明确、运行可控。

---

## 5.5 performance：性能测试

性能测试用于诊断吞吐、延迟、抖动和瓶颈。

性能测试不应只输出 pass / fail，而应输出可解释的诊断指标。

性能测试应明确：

1. 测试目标。
2. 测试规模。
3. 运行时间。
4. warmup 时间。
5. 采样频率。
6. 指标定义。
7. 通过阈值。
8. 失败时如何定位问题。

---

# 6. pytest 使用规范

## 6.1 测试命名

测试函数命名应表达行为，而不是实现细节。

推荐格式：

```text
test_<被测对象>_<场景>_<预期结果>
```

推荐：

```python
def test_scheduler_records_missed_tick_when_reader_is_late() -> None:
    ...


def test_env_parser_uses_default_when_variable_is_missing() -> None:
    ...


def test_metrics_report_counts_failed_reads_without_crashing_run() -> None:
    ...
```

不推荐：

```python
def test_func() -> None:
    ...


def test_scheduler_1() -> None:
    ...
```

---

## 6.2 测试结构

推荐使用 Arrange / Act / Assert 结构：

```python
def test_calculate_miss_rate_returns_zero_when_all_reads_succeed() -> None:
    """Verify miss rate calculation for the fully successful case."""
    # Arrange
    ok_read = 100
    expected_read = 100

    # Act
    miss_rate = calculate_miss_rate(ok_read, expected_read)

    # Assert
    assert miss_rate == 0.0
```

复杂测试可以使用阶段注释：

```python
# Arrange: Build a scheduler with one delayed reader.
...

# Act: Run the scheduler for a short fixed window.
...

# Assert: The scheduler records missed ticks but keeps running.
...
```

---

## 6.3 一个测试只验证一个主要行为

一个测试可以有多个断言，但这些断言应围绕同一个行为。

推荐：

```python
def test_failed_read_is_recorded_as_sample() -> None:
    assert sample.ok is False
    assert sample.error_message is not None
    assert sample.duration_ms >= 0
```

不推荐在一个测试中同时验证：

1. 配置解析。
2. 调度策略。
3. OPC UA 读取。
4. 指标汇总。
5. 报告输出。

这种测试失败时难以定位问题。

# 7. fixture 使用原则

## 7.1 fixture 应表达领域概念

推荐：

```python
@pytest.fixture
def fake_reader() -> FakeSourceReader:
    return FakeSourceReader()
```

不推荐：

```python
@pytest.fixture
def data() -> dict[str, Any]:
    return {"x": 1}
```

除非 `data` 的含义非常明确。

---

## 7.2 fixture 不要过度隐藏逻辑

fixture 应封装重复 setup，而不是把测试关键行为藏起来。

不推荐：

```python
def test_scheduler_works(full_environment: Environment) -> None:
    assert full_environment.result.ok
```

推荐：

```python
def test_scheduler_records_successful_read(
    scheduler_factory: SchedulerFactory,
    fake_reader: FakeSourceReader,
) -> None:
    scheduler = scheduler_factory(reader=fake_reader)

    result = scheduler.run_once()

    assert result.ok_read == 1
```

---

## 7.3 fixture 作用域要谨慎

默认使用 function scope。

只有在明显有收益时才使用 module scope 或 session scope。

原因：

1. function scope 隔离性最好。
2. 大 scope 容易引入测试间状态污染。
3. 性能测试和集成测试尤其容易被共享状态影响。
4. Codex 后续重构时更容易判断测试边界。

---

# 8. mock 与 monkeypatch 原则

## 8.1 优先依赖注入，少 patch

如果代码可以通过参数传入 fake object，就不要 patch 全局对象。

推荐：

```python
scheduler = SourcePollingScheduler(reader=fake_reader)
```

不推荐：

```python
monkeypatch.setattr(module, "reader", fake_reader)
```

依赖注入更清晰，也更适合测试架构边界。

---

## 8.2 使用 monkeypatch 处理环境变量

推荐：

```python
def test_config_reads_target_hz_from_env(monkeypatch: pytest.MonkeyPatch) -> None:
    monkeypatch.setenv("SOURCE_SIM_LOAD_TARGET_HZ", "10")

    config = load_config_from_env()

    assert config.target_hz == 10.0
```

环境变量测试必须覆盖：

1. 未设置时默认值。
2. 设置合法值。
3. 设置非法值。
4. 兼容旧变量。
5. 新旧变量同时存在时的优先级。
6. 类型转换错误。

---

## 8.3 mock 应尽量带规格

如果必须使用 mock，优先使用 `create_autospec()` 或明确协议类型，避免普通 `Mock` 过于宽松。

推荐：

```python
from unittest.mock import create_autospec

reader = create_autospec(SourceReaderProtocol, instance=True)
reader.read_once.return_value = sample
```

不推荐：

```python
reader = Mock()
reader.read_once.return_value = sample
```

普通 `Mock` 太宽松，函数名写错、参数传错也可能测试通过。

---

# 9. 测试数据原则

## 9.1 测试数据要小而明确

推荐：

```python
points = [
    PointDefinition(path="IED1.LD0.MMXU1.TotW", data_type="float"),
    PointDefinition(path="IED1.LD0.MMXU1.Hz", data_type="float"),
]
```

不推荐在单元测试里构造几十上百个点位，除非测试目标就是规模。

---

## 9.2 使用 builder / factory 构造复杂对象

如果对象构造复杂，应提供测试 builder。

```python
def make_polling_job_spec(
    source_id: str = "source-1",
    interval_s: float = 0.1,
    timeout_s: float = 1.0,
) -> PollingJobSpec:
    return PollingJobSpec(
        source_id=source_id,
        interval_s=interval_s,
        timeout_s=timeout_s,
    )
```

测试里只覆盖关心字段：

```python
spec = make_polling_job_spec(interval_s=0.05)
```

---

## 9.3 不要复制大段真实配置

真实配置适合放在 integration 或 e2e 测试中。

单元测试应使用最小配置。

---

# 10. 异步测试原则

## 10.1 异步测试必须显式标注

```python
@pytest.mark.asyncio
async def test_scheduler_runs_async_reader() -> None:
    ...
```

---

## 10.2 避免真实长时间 sleep

不推荐：

```python
await asyncio.sleep(5)
```

推荐：

1. 使用短时间窗口。
2. 注入 fake clock。
3. 把纯时间计算逻辑拆出来单测。
4. 对必须真实等待的集成测试使用 marker 标记为 slow。

---

## 10.3 异步测试必须清理后台任务

```python
task = asyncio.create_task(scheduler.run())

try:
    ...
finally:
    task.cancel()
    with contextlib.suppress(asyncio.CancelledError):
        await task
```

避免测试结束后遗留后台 task。

---

# 11. 调度器测试原则

调度器测试要区分三类逻辑：

1. 纯计算逻辑。
2. 单次 tick 行为。
3. 长时间运行行为。

---

## 11.1 纯计算逻辑单测

适合测试：

1. next deadline 计算。
2. missed tick 判断。
3. absolute / relative 模式差异。
4. burst / stagger offset 计算。
5. interval 转换。
6. tolerance 判断。

这些测试不应启动真实 reader。

---

## 11.2 单次 tick 测试

验证：

1. reader 被调用。
2. 成功事件被记录。
3. 失败事件被记录。
4. timeout 被转换为错误事件。
5. 调度器不会吞掉错误。
6. 单个 reader 失败不会导致整体崩溃，除非设计要求如此。

---

## 11.3 长时间运行测试

长时间运行测试应尽量少。

验证：

1. 调度器能持续运行。
2. stop signal 能停止。
3. task cancellation 能清理。
4. 不会因单个 reader 失败而整体崩溃。

这类测试应加 marker：

```python
@pytest.mark.integration
@pytest.mark.slow
```

---

# 12. 性能测试原则

## 12.1 性能测试的目标

性能测试的目标不是输出更多调用栈，而是定位瓶颈。

对于 simulator / OPC UA / scheduler 类性能测试，应尽量把链路拆开：

```text
scheduler wake lag
client read call duration
decode duration
post-process duration
reader tick duration
client period jitter
server update period
per-server success/miss/error
```

---

## 12.2 性能测试不要只看一个指标

至少区分：

1. read call duration：协议读取调用耗时。
2. tick duration：完整 reader tick 耗时。
3. client period：客户端相邻读取间隔。
4. jitter：客户端采样抖动。
5. miss rate：未完成预期读取比例。
6. achievement ratio：达到预期读取次数比例。
7. error rate：读取错误比例。
8. post-processing duration：收到数据后的处理耗时。

---

## 12.3 指标命名必须清楚

推荐：

```text
read_call_p95_ms
read_call_p99_ms
tick_p95_ms
tick_p99_ms
client_period_p95_ms
client_jitter_p95_ms
post_process_p95_ms
miss_rate
achievement_ratio
error_rate
```

不推荐：

```text
p95
duration
cost
time
```

因为这些名字无法说明测量对象。

---

## 12.4 性能测试应输出 summary，而不是海量调用栈

性能测试失败时，应优先输出：

1. 测试参数。
2. 通过 / 失败指标。
3. top N 最大 gap。
4. top N 最慢 read。
5. top N 最慢 post-process。
6. 每个 server 的统计。
7. 每个 reader 的统计。
8. 关键分位数。
9. 失败原因推断。

只有在需要时才输出详细调用栈或 profiler dump。

---

## 12.5 分离严格断言和诊断数据

严格断言示例：

```python
assert metrics.miss_rate <= max_miss_rate
assert metrics.client_period_pass_ratio >= min_client_period_pass_ratio
```

诊断数据示例：

```text
read_call_p95_ms
tick_p95_ms
jitter_p95_ms
post_process_p95_ms
top_gap_reason
```

这样失败时可以判断问题更可能出在：

1. server 慢。
2. client event loop 抖动。
3. 解码慢。
4. 后处理慢。
5. burst 调度造成集中冲突。
6. process / coroutine 模式差异。
7. 协议库调用成本过高。

---

# 13. marker 策略

推荐使用 pytest marker 对测试分类：

```python
@pytest.mark.unit
@pytest.mark.integration
@pytest.mark.smoke
@pytest.mark.e2e
@pytest.mark.performance
@pytest.mark.slow
@pytest.mark.requires_opcua
@pytest.mark.requires_open62541
```

常用命令：

```bash
pytest -m unit
```

```bash
pytest -m "not slow"
```

```bash
pytest -m performance -s -v
```

```bash
pytest -m requires_opcua
```

marker 的作用是让 Codex 和开发者可以快速选择合适的测试范围，而不是每次都盲目跑全量测试。

# 14. Codex 修改代码后的测试选择策略

Codex 不应每次都盲目跑全量测试。应根据改动范围选择测试。

---

## 14.1 修改纯函数

例如指标计算、路径拼接、配置解析。

应运行：

```bash
python -m py_compile <changed-files>
ruff check <changed-files>
mypy <changed-files-or-package>
pytest <related-unit-test-file> -q
```

---

## 14.2 修改调度器

应运行：

```bash
pytest tests/unit/<related-scheduling-tests> -q
pytest tests/integration/<related-scheduling-tests> -q
```

如果涉及异步调度或性能采样，还应跑最小 profile 测试。

## 14.3 修改 adapter

应运行：

```bash
pytest <adapter-unit-tests> -q
pytest <adapter-integration-tests> -q
```

如果 adapter 涉及外部协议，应优先跑 fake server 或 local server 测试，不要直接依赖真实环境。

## 14.4 修改配置解析

应运行：

```bash
pytest <config-tests> -q
```

必须覆盖：

1. 默认值。
2. 合法值。
3. 非法值。
4. 兼容变量。
5. 优先级。
6. 类型转换错误。

## 14.5 修改性能测试代码

应运行：

```bash
python -m py_compile <changed-test-file>
pytest <changed-test-file> -s -v
```

同时检查：

1. 指标名是否准确。
2. summary 是否可读。
3. 失败时是否能定位问题。
4. 是否误把诊断指标当成修复手段。
5. 是否保持历史字段兼容。
6. 是否避免无意义的大量调用栈输出。


同时检查：

1. `__init__.py` 是否存在。
2. 包导入是否符合项目结构。
3. 测试是否通过正确路径导入源码。
4. 是否引入 namespace package 误识别问题。
5. 是否破坏已有 CLI 或入口脚本。

# 15. 回归测试原则

如果修复 bug，必须新增或修改测试来复现 bug。

推荐流程：

1. 先写一个失败测试，复现问题。
2. 修改实现。
3. 确认测试通过。
4. 跑相关回归测试。
5. 在提交说明中说明新增测试覆盖了哪个 bug。

Codex 必须避免只改实现、不补回归测试。

---

# 16. 删除或重构测试的原则

Codex 不得随意删除测试。

允许删除测试的情况：

1. 被测功能已删除。
2. 测试重复且没有额外覆盖价值。
3. 测试验证的是错误旧行为。
4. 测试长期 flaky，且已有更稳定替代测试。
5. 测试依赖过时文件或过时接口。
6. 测试与当前架构边界明显冲突。

删除测试时必须说明：

1. 删除原因。
2. 是否有替代测试。
3. 覆盖范围是否损失。
4. 是否需要新增其他测试补位。

---

# 17. flaky 测试处理原则

flaky 测试不能简单放宽断言。

应先判断原因：

1. 时间窗口过短。
2. 依赖真实 sleep。
3. 依赖测试顺序。
4. 共享状态未清理。
5. 端口冲突。
6. 异步 task 未取消。
7. 性能阈值过于接近机器极限。
8. 测试混合了行为验证和性能诊断。
9. 依赖当前机器负载。
10. 依赖外部服务状态。

处理方式：

1. 缩小测试目标。
2. 拆分纯逻辑测试和真实运行测试。
3. 使用 fake clock 或 fake reader。
4. 增加 warmup。
5. 使用动态端口。
6. 清理后台 task。
7. 将性能阈值和诊断指标分离。
8. 将慢测试标记为 slow 或 performance。

不要无脑提高容忍度。

# 18. 断言原则

## 18.1 断言要表达业务含义

不推荐：

```python
assert result == 1
```

推荐：

```python
assert result.ok_read == 1
assert result.miss_rate == 0.0
```

---

## 18.2 浮点数使用 approx

```python
assert miss_rate == pytest.approx(0.1)
```

---

## 18.3 异常断言要检查消息

```python
with pytest.raises(ValueError, match="expected_read"):
    calculate_miss_rate(ok_read=1, expected_read=0)
```

---

## 18.4 不要只断言“不报错”

不推荐：

```python
def test_run() -> None:
    run_pipeline()
```

推荐：

```python
def test_run_pipeline_writes_expected_snapshot() -> None:
    result = run_pipeline()

    assert result.snapshot_count == 1
    assert result.error_count == 0
```

---

# 19. 测试代码文档要求

复杂测试文件必须有模块级 docstring。

尤其是：

1. 性能测试。
2. 集成测试。
3. e2e 测试。
4. 涉及多模式参数化的测试。
5. 涉及多个环境变量的测试。
6. 涉及外部协议的测试。
7. 涉及调度、异步、进程、端口、临时服务的测试。

文件头应说明：

1. 测试目标。
2. 测试范围。
3. 测试模式。
4. 指标定义。
5. 断言含义。
6. 运行方式。
7. 依赖条件。
8. 哪些失败属于性能瓶颈，哪些失败属于功能错误。

# 20. 参数化测试原则

适合参数化的场景：

1. 多个合法配置值。
2. 多个非法配置值。
3. 多种调度模式。
4. 多种 reader mode。
5. 多种 read schedule。
6. 多种数据类型转换。
7. 多种环境变量组合。

示例：

```python
@pytest.mark.parametrize(
    ("raw_value", "expected"),
    [
        ("true", True),
        ("1", True),
        ("false", False),
        ("0", False),
    ],
)
def test_parse_bool_env_accepts_supported_values(
    monkeypatch: pytest.MonkeyPatch,
    raw_value: str,
    expected: bool,
) -> None:
    monkeypatch.setenv("FEATURE_ENABLED", raw_value)

    assert parse_bool_env("FEATURE_ENABLED") is expected
```

参数化测试要避免过度组合爆炸。必要时拆成多个更清晰的测试。

# 21. 测试代码质量要求

测试代码也是工程代码。

测试代码必须遵守：

1. ruff。
2. 基本类型注解。
3. 清晰命名。
4. 不残留临时调试代码。
5. 不残留无用 fixture。
6. 不残留复制粘贴的大段重复代码。
7. 不使用不可解释的魔法数字。
8. 不把测试写成脚本式流水账。
9. 不依赖测试执行顺序。
10. 不依赖当前机器环境的偶然状态。

性能测试可使用 `print` 或 report 输出，但必须是结构化 summary，不是临时调试输出。

# 26. 最终建议

对当前项目，建议采用以下原则：

1. 正式源码测试放在根目录 `tests/`。
2. 独立工具测试可以放在 `tools/<tool_name>/tests/`。
3. 不要把测试文件放进 `src/` 源码目录。
4. 不要把所有测试平铺在 `tests/` 下。
5. 按 `unit / integration / smoke / e2e / performance` 分层。
6. 分层后再按源码模块路径组织。
7. Codex 修改代码后，先跑最小相关测试，再按影响范围扩大。
8. 性能测试必须输出可诊断 summary，而不是堆调用栈。
9. 修 bug 必须补回归测试。
10. flaky 测试必须定位原因，不要简单放宽阈值。
